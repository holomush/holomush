# Web Client Guidelines

Instructions for working with the HoloMUSH web client (SvelteKit).

## Tech Stack

| Layer | Technology |
| --- | --- |
| Framework | SvelteKit 2, Svelte 5 |
| Components | shadcn-svelte (nova style) on bits-ui headless primitives |
| Styling | Tailwind CSS v4 (layout/spacing), CSS custom properties (theming) |
| Build | Vite, pnpm |
| Protocol | ConnectRPC (gRPC-Web) via `@connectrpc/connect-web` |
| Testing | Playwright (E2E), vitest (unit) |

## Theme System

### Architecture

The theme system uses CSS custom properties with runtime switching:

```text
app.css (@theme)            → registers tokens with static default-dark values (build-time)
themeStore (inline style)   → overrides --color-* on .app-root (runtime)
scoped CSS / Tailwind utils → read var(--color-*) — same names everywhere
```

### Tailwind v4 + Runtime Theming

**CRITICAL:** Tailwind v4's `@theme` block compiles `var()` references at
**build time**, not runtime. Never use `var()` inside `@theme` for values
that change at runtime.

```css
/* WRONG — var(--my-primary) resolves at build time to the :root value.
   Runtime changes to --my-primary on a descendant element are invisible. */
@theme {
  --color-primary: var(--my-primary);
}

/* RIGHT — static value registers the token. Runtime inline styles
   on .app-root override --color-primary via CSS cascade. */
@theme {
  --color-primary: #ff7043;
}
```

The theme store sets `--color-primary: <value>` on `.app-root` as an inline
style. CSS custom property inheritance ensures all children see the override.

### CSS Variable Naming

Two prefixes, one rule:

| Prefix | Used for | Example |
| --- | --- | --- |
| `--color-*` | Everything except MUSH message colors | `--color-primary`, `--color-background`, `--color-card` |
| `--mush-*` | MUSH message colors only (11 tokens) | `--mush-say-speaker`, `--mush-pose-actor`, `--mush-system` |

- `--color-*` tokens are consumed by both Tailwind utilities (`bg-primary`)
  and scoped CSS (`var(--color-primary)`)
- `--mush-*` tokens are consumed only by terminal renderer components
- `--radius` is the sole unprefixed token (not a color)

**MUST NOT** use bare variable names (`--primary`, `--background`) for theme
tokens. Always use `--color-*` prefix so the same name works in both Tailwind
utilities and scoped CSS without an indirection layer.

### Theme Files

Themes are JSON files in `src/lib/theme/`:

```json
{
  "name": "default-dark",
  "colors": {
    "say.speaker": "#4fc3f7",
    "primary": "#ff7043",
    "background": "#1a1210"
  }
}
```

The JSON keys use semantic names (no CSS prefixes). The `themeToCssVars()`
function maps keys to CSS variables:

- MUSH tokens (`say.speaker`) → `--mush-say-speaker`
- Everything else (`primary`) → `--color-primary`

### Adding a New Theme Token

1. Add the key to `ThemeColors` interface in `src/lib/theme/types.ts`
2. Add the value to all 4 theme JSON files
3. Add a static default to `@theme` in `app.css`
4. If it's a MUSH message color, add to `MUSH_TOKENS` set in `themeStore.ts`

### Terminal Black Background Override

A boolean toggle that overrides `--color-background` and
`--color-input-background` to pure black on the terminal container only.
Controlled by `ThemePreferences.terminalBlackBackground`.

## Component Organization

### shadcn Components (`src/lib/components/ui/`)

Generated by shadcn-svelte CLI. Used for forms, buttons, dialogs, layout
structure. These consume `--color-*` tokens via Tailwind utility classes.

```bash
pnpm dlx shadcn-svelte@latest add <component-name>
```

### Custom Components

Terminal renderers and game-state displays are custom. They MUST follow
shadcn conventions:

- Use `cn()` utility from `$lib/utils` for conditional classes
- Pull colors from `var(--color-*)` or `var(--mush-*)` tokens
- Expose a `class` prop for extension

### What Goes Where

| Component type | Location | Uses shadcn? |
| --- | --- | --- |
| Forms, buttons, cards, dialogs | shadcn `ui/` components | Yes |
| Terminal container, sidebar split | shadcn Resizable, Sheet | Yes |
| Message renderers (say, pose, etc.) | `terminal/*Renderer.svelte` | No (custom) |
| Game state (room, exits, presence) | `sidebar/*.svelte` | No (custom) |
| Command input, status bar | `terminal/*.svelte` | No (custom) |

## Svelte 5 Notes

- `<script context="module">` is removed in Svelte 5. All imports go in
  the instance `<script lang="ts">` block.
- Use `$props()`, `$state()`, `$derived()`, `$effect()` runes.
- Use `{@render children()}` instead of `<slot />`.
- Use `onclick={handler}` instead of `on:click={handler}`.

## Form Requirements

- All form inputs MUST have `name` attributes for Playwright E2E testability
- Buttons that submit forms MUST have `type="submit"`

## Auth Guards

SvelteKit `load()` runs BEFORE `onMount()`. Session restoration from
`sessionStorage` must happen in `load()`, not `onMount()`, or auth guards
will redirect on page reload.
