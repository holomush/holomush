### Decision — manifest `history_scope` (R4-A)

**Use `history_scope: custom`, NOT `history_scope: channel`.** `history_scope` is a CLOSED enum accepting only `{grid, scene, custom}` (`internal/plugin/manifest.go:381`); unknown values are rejected at manifest validation / plugin load (`internal/plugin/manifest.go:557`), so a `channel` value would make `core-channels` FAIL TO LOAD once `emits: [channel]` is present — and the 01-09 whole-system census (which asserts the plugin loads) would fail. **`custom` is the semantically correct fit:** the field's own doc (`manifest.go:143`) defines `custom` as "plugin owns visibility via QueryHistory RPC" — exactly channels' membership-gated, per-channel history model (01-06 `PluginAuditService.QueryHistory` authorizes reads against the plugin-owned membership store; visibility is NOT delegated to the host `grid`/`scene` buckets). `grid` (all grid observers) and `scene` (scene participants — channels are location-independent, no scene, CHAN-01) are both semantically wrong. This is a ZERO host change: `history_scope` has no runtime consumer outside the validator (declarative INV-PRIVACY-7 bucket only), so `custom` imposes no behavior that could break per-channel membership-gated history or the 30-day retention/prune — retention is driven by the plugin's own `retention_window`/`prune_interval` config (01-07), not by `history_scope`. The committed `schemas/plugin.schema.json` declares `history_scope` as a bare string (no value enum), so no schema regeneration is needed.

**Why not add `channel` to the enum — and is `scene` a bug?** Neither. Per the history-scope privacy spec (`docs/superpowers/specs/2026-05-17-history-scope-privacy-design.md` §I-PRIV-7): `grid` and `scene` are the host's TWO BUILT-IN privacy models — the host genuinely implements their replay-visibility floors in `streamScopeFloor`, keyed off the `location:<id>` / `scene:<id>` SUBJECT patterns (spec §Scene-join floor), so they are host privacy semantics, not plugin vocabulary that leaked in. `custom` is the spec's declared value for **plugin-owned subjects whose semantics diverge from the grid/scene model**, and the spec NAMES channels as exactly such a subject (§"Plugin-owned subjects (channels, etc.) MAY adopt different semantics per their own design but MUST declare them explicitly"). Adding `channel` to the enum would be a CATEGORY ERROR: it would assert a host-built-in "channel privacy model" (with its own floor machinery) that does not and must not exist — channels own their visibility via their own `QueryHistory`. So `custom` is spec-mandated, not merely convenient. (Bonus: core-channels is the FIRST plugin to adopt `history_scope: custom` — see 01-09 Task 3, which uses that to bind the currently-unbound INV-PRIVACY-7.)
