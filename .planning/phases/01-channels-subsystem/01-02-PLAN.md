<objective>
Close the net-new live-delivery substrate gaps (Landmine 2) so `core-channels` — the FIRST consumer of session-stream subscription — can deliver live messages. Cross-AI review (Codex, source-grounded) confirmed three of these are genuine blockers/security gaps, not just "unserved" stubs; each is now first-class plan work (HIGH-1/2/3):

1. **Session-establishment path:** add the binary-SDK `SessionStreamsHandler` hook so a binary plugin's `QuerySessionStreams` RPC routes to plugin code (today `pluginServerAdapter` embeds `UnimplementedPluginServiceServer` → `codes.Unimplemented`). Lua parity already exists (`on_session_subscribe`, `internal/plugin/lua/host.go`), so this CLOSES a binary/Lua gap rather than opening one. **AND (review HIGH-1)** fix `Manager.QuerySessionStreams` so it does not DROP the dot-style channel streams the plugin returns: `isValidStreamName` (`internal/plugin/manager.go:1496-1506`) currently requires a colon and drops non-colon streams (`:1566-1571`), contradicting the dot-eradication in `internal/grpc/server.go:707-733` + `internal/eventbus/qualify.go`. Make it accept dot-relative / full `events.` subjects that `eventbus.Qualify` accepts.
2. **Mid-session join/leave path:** serve the `stream.subscription` host capability (`StreamSubscriptionService.AddSessionStream`/`RemoveSessionStream` — proto-declared but server-stubbed as `codes.Unimplemented`, tracked `holomush-l6std`) by wiring it to the existing host `StreamRegistry` (`internal/grpc/stream_registry.go`), and add the binary-SDK client + `*Aware` interface so a plugin can call it. **This entails two review findings that are net-new work, not free:**
   - **(HIGH-2) Implement `LIVE_ONLY` end-to-end.** `SessionStreamRegistry.AddStreamWithMode` (`stream_registry.go:190-199`) today REJECTS any mode ≠ `ReplayModeFromCursor` with `REPLAY_MODE_NOT_SUPPORTED`; serving the RPC alone does NOT deliver "no history flood." Make `AddStreamWithMode` ACCEPT `ReplayModeLiveOnly` (additive; keep FROM_CURSOR for scenes, keep rejecting BoundedTail if still unsupported) and PROVE with a subscribe-loop test that a mid-session-added stream delivers only new events. The no-history-flood guarantee is available because `SetFilters` preserves the live consumer's start policy on filter rotation (`internal/eventbus/subscriber.go:348-351`) — the registry accepting the mode is the fix; the test locks the guarantee.
   - **(HIGH-3, SECURITY) Add an instance-level concrete-stream authorization guard.** Both subscription RPCs map to `write` on `stream` (`internal/plugin/hostcap/descriptor.go:154-157`) and the seed default-permits declared plugins at `resource == "stream:*"` at the TYPE level (`internal/access/policy/seed.go:423-427`); the seed comment (`:443-444`) states stream WRITES are "gated solely by the type-level capability" — so without a per-call guard, ANY plugin granted `stream.subscription` could ask the host to subscribe an active session to ARBITRARY streams (including `events.<game>.system.*`, audit/crypto namespaces, or another plugin's domain). Mirror `stream.history`'s in-handler concrete-stream check (`internal/plugin/hostcap/servers.go:829-854` → `pluginauthz.AuthorizeStreamRead`): require a domain-RELATIVE ref, qualify the requested stream, reject wildcards and forbidden namespaces, and verify the stream is in a namespace the calling plugin OWNS (its declared `emits` domain). Add the SDK client + `*Aware` interface so a plugin can call the served capability.
     - **(R2-A) Canonical input form = domain-RELATIVE `channel.<id>`.** The cross-AI re-review (Codex R2) found 01-02 and 01-08 disagreed on the subject form: `AuthorizeStreamSubscribe` (mirroring `AuthorizeStreamRead`, `pluginauthz/streamread.go:43-55`) REJECTS a pre-qualified `events.` subject (`STREAM_NOT_RELATIVE`), but 01-08 was passing a FULL `events.<game>.channel.<id>` — the guard would reject the plugin's OWN join. Resolution (verified against source): pick ONE canonical input form = **domain-RELATIVE `channel.<id>`**, and let the guard `eventbus.Qualify` it, exactly like the read guard. This is safe because BOTH downstream registry paths already qualify idempotently: the mid-session ctrl path `server.go::applyFilterCtrl` (`internal/grpc/server.go:1382`) calls `toSubject`→`Qualify`, and the establishment path `computeInitialFilters`→`toSubject` (`:707-738`) does too. The SDK client forwards the relative ref; the server forwards `req.GetStream()` (relative) to `AddStreamWithMode` exactly as the read path forwards `req.GetStream()` to `ReplayTail`. 01-08 is updated in lockstep to pass the relative form (a `relativeChannelStream(id)` helper), NOT the full-subject `dotStyleChannelSubject`.
     - **(R2-B) Forbidden-namespace protection is IN-HANDLER, not via seed forbids.** The audit/system/crypto stream denies AND the concrete stream permit are ALL `action in ["read"]` (verified `internal/access/policy/seed.go:236,253,274,298,448`; the comment at `:443-444` explicitly says "stream writes (stream.subscription) remain gated solely by the type-level capability"). A NEW `seed:plugin-stream-subscribe` `write` permit is therefore NOT carved back by any of those read-only forbids. So the load-bearing forbidden-namespace control MUST be in the handler: `AuthorizeStreamSubscribe` explicitly REJECTS the `system`/`audit`/`crypto` domain segments AND enforces the owned-namespace fence (the qualified subject's `<domain>` MUST be one of the plugin's declared `emits` domains — reuse the `event_emitter.go::subjectNamespace` + `declaresEmitNamespace(Manifest.Emits, ns)` pattern at `:98-110`). The owned-namespace fence already excludes system/audit/crypto for any plugin that doesn't own them, but the explicit forbidden-namespace rejection is made mandatory AND independently tested. The `seed:plugin-stream-subscribe` write permit remains only as the type/instance capability grant; optional write-specific `forbid(... action in ["write"] resource is stream ...)` seed policies MAY be added as defense-in-depth, but the in-handler rejection is the mandatory, independently-tested control.

Purpose: `core-channels` needs both paths — QuerySessionStreams for connect/reconnect subscription, AddStream/RemoveStream(LIVE_ONLY) for `channel join`/`leave` during a live session — and both must be SAFE and non-flooding.
Output: SDK `SessionStreamsHandler` + `StreamSubscription` client/Aware; dot-subject-accepting `QuerySessionStreams`; LIVE_ONLY-honoring registry; served `stream.subscription` capability WITH a concrete-stream authz guard + `seed:plugin-stream-subscribe`.

**abac-reviewer GATE (per orchestrator directive):** Task 2/3 touch `internal/access/policy/seed.go` and `internal/plugin/pluginauthz/` — the `abac-reviewer` domain gate MUST fire before push for this plan (CLAUDE.md § Pre-Push Review Gates; `/holomush-dev:review-abac`). Every host-side change here is SHARED substrate and MUST be ADDITIVE — preserve existing scene session-stream + stream.history behavior (add the FROM_CURSOR regression assertion and a stream.history-unchanged assertion).

**SCOPING RISK (flagged per orchestrator directive):** This is genuine host+SDK substrate plumbing, not plugin domain code. It is BOUNDED and well-scaffolded — the proto (`api/proto/holomush/plugin/host/v1/stream.proto`), the capability token (`internal/plugin/capability_vocab.go`), the auto-generated Lua bridge (`internal/plugin/luabridge/bindings_gen.go`), and the host `StreamRegistry` implementation all already exist; only the server bodies + binary-SDK client/hook + the LIVE_ONLY-accept + the concrete-stream authz guard are missing. If Task 2 (serving `stream.subscription`, i.e. mid-session mutation) proves larger than budgeted, the DELIVERY FLOOR is Task 1 alone: session-establishment delivery via `QuerySessionStreams` (with the HIGH-1 dot-subject fix) still gives every member their channel streams at connect/reconnect. Mid-session live `join` would then degrade to "subscribed on next session-stream refresh" rather than instantly — a UX degradation, NOT a dropped requirement. Do NOT silently drop mid-session delivery; if it cannot land, note the degradation explicitly in the SUMMARY and file a follow-up bead against `holomush-l6std`. The HIGH-3 authz guard is NOT descopable: `stream.subscription` MUST NOT be served without its concrete-stream gate.
