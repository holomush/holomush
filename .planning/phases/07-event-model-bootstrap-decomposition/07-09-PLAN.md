#### The shape of the problem (verified live)

`runCore` is a ~900-line straight-line wiring routine that must construct all 17
subsystems **before** `orch.StartAll`. `store.DatabaseSubsystem.Start`
(`subsystem.go:50-78`) is what **creates** the pool (`s.pool = es.Pool()` at `:65`)
and resolves the gameID (`InitGameID`, `:67`) — so there is no live pool to hand
out before it runs. That is the whole mechanism: **wiring happens before starting,
and some wiring needs a started resource.** Hence the pre-starts.

D-09 is locked (*"constructors take handles/providers, not resolved live values;
resolution happens at `Start` time"*), so the only compliant shape is: **the reads
move to where the graph guarantees the resource is up.**

#### The settled design: hoist the wiring's RESOLUTION LOGIC into a memoized `cryptoWiring`, resolved at Start — the block's subsystem CONSTRUCTIONS stay behind

`core.go:705-1060` — the crypto/admin wiring block — is **17 of the ~20 reads**
in one contiguous region. Its resolution logic is hoisted into a memoized
builder in `package main`:

```
type cryptoWiring struct { /* the block's shared resolved outputs — table below */ }
// resolveCryptoWiring returns a func() (*cryptoWiring, error) memoized with
// sync.Once, closing over dbSub/authSub/abacSub/eventBusSub and cfg.
```

- ⚠️ **"Hoisted whole" / "the block's body moves verbatim" is RETRACTED (cross-AI
  round 7, BLOCKER 2 — the instruction was not implementable literally).** The
  `:705-1060` block itself CONSTRUCTS three lifecycle subsystems —
  `cryptoPolicySub` (`core.go:738`), `rekeyCheckpointSweepSub` (`:774`), and
  `cryptoChainVerifierSub` (`:1053`) — and every subsystem must exist and be
  registered **before** `orch.StartAll` (`:1081`). A memoized builder first
  invoked by one of those subsystems' own `Start` cannot also create that
  subsystem. The corrected split:
  - **The block's RESOLUTION LOGIC moves** into the memoized builder: every
    `dbSub.Pool()` / `authSub.Hasher()` / `abacSub.Resolver()` /
    `eventBusSub.Conn()` / `eventBusSub.Publisher()` read, the repo/emitter/
    manager/handler construction those reads feed, `provisionBootKEKProvider`,
    `buildRekeyWiring`, `buildReadStreamWiring`, and the degraded-path WARNs.
    Those reads do not change at all inside the moved logic — they simply now
    execute after those subsystems have started.
  - **The block's SUBSYSTEM CONSTRUCTIONS stay on runCore's straight-line path**,
    each constructed with a provider-shaped config (§ "The per-consumer provider
    signatures") whose closure projects off the memoized builder. The
    constructors allocate nothing and read no live value — D-09-clean — so they
    are legal before `StartAll`.
- **The `cryptoWiring` struct fields — enumerated, with the pre-`StartAll`
  constructor that consumes each (round 7, BLOCKER 2's required carve-out):**

  | `cryptoWiring` field (shape) | Consuming pre-`StartAll` constructor | Via provider |
  |---|---|---|
  | `policyEmitDeps policy.EmitDeps` | `cryptoPolicySub` (`core.go:738`-ish) | `EmitDepsProvider` |
  | `checkpointRepo *dek.CheckpointRepo` + `checkpointAuditEmitter dek.AuditEmitter` | `rekeyCheckpointSweepSub` (`:774`-ish) | `DepsProvider` |
  | `chainRepo chain.Repo` | `cryptoChainVerifierSub` (`:1053`-ish) | `RepoProvider` |
  | `chainHandlers []chain.Handler` | `cryptoChainVerifierSub` | `HandlersProvider` |
  | `adminHandlers socket.Handlers` (Authenticate/Approve/ResetTOTP/Rekey/ReadStream) | `adminSub` | `HandlersProvider` |
  | `rekeyManager dek.Manager` + the Coordinator/invalidation inputs (`invalidation.Deps` fields incl. Conn; `coordHolderPtr`) | `grpcSubsystem` construction | direct `func() (*cryptoWiring, error)` |

  Field names MAY drift at implementation; the SET may not — a wiring output a
  consumer needs that no field carries is a stop-and-report, and a field with no
  consumer is dead weight to delete.
- **Consumers take a provider, not a value — with CONSUMER-OWNED provider
  types.** ⚠️ Rev 7 said each consumer config "takes `func() (*cryptoWiring,
  error)` (or a narrow accessor off it)" — **not package-realizable for 4 of the
  5 consumers** (cross-AI round 5, BLOCKER): `cryptoWiring` is a `package main`
  type that `internal/admin/policy`, `internal/eventbus/crypto/dek`,
  `internal/eventbus/audit/chain`, and `internal/admin/socket` cannot name.
  Retracted. The settled per-consumer signatures are in § "The per-consumer
  provider signatures" below; every provider is resolved **in the consumer's own
  `Start`**, and the production closures — defined in `runCore`, projecting
  fields off the single memoized `resolveCryptoWiring` — are the only code that
  names `*cryptoWiring`.
- **Fatal stays fatal.** `buildRekeyWiring`'s error (`core.go:936`),
  `BOOT_KEK_REQUIRED` (`:816`) and
  `CRYPTO_DEK_MANAGER_CACHE_ACCESSOR_NOT_SATISFIED` (`:961`) surface from the
  memoized builder into the first consumer's `Start`, which returns them →
  `StartAll` aborts the boot. Identical outcome to today's `return err` from
  `runCore`, just relocated.
- **The memoized closure caches BOTH the result and the error (codex round 6,
  suggestion — adopted).** The `sync.Once` body stores `(*cryptoWiring, error)`
  into captured variables; every subsequent resolution returns the SAME pair. A
  failed build is therefore **never retried within one process** — deliberately:
  `StartAll` aborts on the first consumer's error anyway, and a retrying
  memoizer would let consumers observe divergent wiring. State this no-retry
  contract in `resolveCryptoWiring`'s doc comment.
- **WARN-and-continue stays WARN-and-continue.** Every degraded path inside the
  block (`:831`, `:851`, `:872`, `:881`, `:939`, `:975`, `:978`, `:1036`) is
  **inside the hoisted body** and is untouched.
- **This is not a second ordering authority (D-10).** The builder does no
  ordering — it is lazy memoization. Ordering remains `topoSort` + `DependsOn`,
  exactly as D-10 requires. It is not a latch, a countdown, or a barrier.

#### ⚠️ THE RULE that makes this safe — state it in code, pin it with a test

> **Every subsystem whose config holds a `cryptoWiring` provider MUST declare the
> wiring's full dependency set — `DependsOn(SubsystemDatabase, SubsystemAuth,
> SubsystemABAC, SubsystemEventBus)` — because the FIRST consumer to resolve the
> provider is the one that builds it.**

A consumer missing an edge could start before `Auth` and trigger the build →
`authSub.Hasher()` panics. This is the one sharp edge of the design and it MUST
be blunted mechanically: add a test asserting each wiring consumer's `DependsOn`
set is a **superset** of the wiring's dep set. Under Wave B (07-11) the resolution
point becomes `Prepare`, which the global barrier makes safer still.

#### ⚠️ THE RULE MUTATES `chain.VerifierSubsystem`'s DEPS. This is correct — and it forbids the reverse edge forever.

**Read this before 07-10 runs; it is the cross-plan consequence of THE RULE.**

`chain.VerifierSubsystem` is one of the five consumers, so after this plan its
`DependsOn()` grows from `[SubsystemDatabase]`
(`internal/eventbus/audit/chain/verifier_subsystem.go:67-69`, true on `main`
today) to a superset including **`SubsystemEventBus`**.

**That is not an artifact of THE RULE being over-broad — it is real, and forced by
data flow.** Verified live:
- `chainHandlers` (`core.go:1047-1052`) appends `readStreamW.AuditChainHandler`
  iff `readStreamW.Handler != nil`.
- `buildReadStreamWiring` **hard-requires** a non-nil `AuditPublisher`
  (`cmd/holomush/readstream_wiring.go:118` — nil ⇒ empty wiring returned).
- `auditPublisher` is `eventbus.NewRenderingPublisher(eventBusSub.Publisher(), …)`
  (`core.go:735`).

⇒ **Building the verifier's own handler set reads `eventBusSub.Publisher()`.** The
bus must be up before the verifier can know which chains it is verifying.

**⚠️ CONSEQUENCE — the edge `EventBus → CryptoChainVerifier` is now IMPOSSIBLE.**
Adding it closes `EventBus → CryptoChainVerifier → EventBus`, which `topoSort`
(`internal/lifecycle/orchestrator.go:108`) detects and **refuses startup** on.
**Rev 4 of 07-10 planned exactly that edge**, proving "no cycle" against the
verifier's **pre-this-plan** deps. It is retracted there; this note exists so the
constraint is visible from **this** side too, since a reader of 07-09 alone would
not know a sibling plan wants the reverse edge. See `07-10-PLAN.md`
`<settlements>` § MEDIUM-11 for the full derivation, and note it is **07-10 Task
4** — a `topoSort` acyclicity test over the real production graph — that mechanically
prevents any future revision from reintroducing this.

**⚠️ THE RULE's superset test CANNOT catch this.** It passes with the cycle fully
present: every consumer *does* declare ⊇ the required set. Green test suite,
unbootable binary. Do not treat it as cycle coverage.
