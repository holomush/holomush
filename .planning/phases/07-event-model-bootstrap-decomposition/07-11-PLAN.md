### D-13.1 — two-phase rollback semantics. SETTLED. **There is no in-tree analog; this is genuine design work.**

`Orchestrator` has exactly one phase today; `StopAll` walks only `startOrder`
(`orchestrator.go:81-96`). "Activate fails after N prepared" has no precedent
anywhere in the repo. The design:

**1. `StartAll` = two full sweeps, not per-subsystem Prepare-then-Activate.**
- Sweep 1: `Prepare` every subsystem in `topoSort` order.
- Sweep 2: `Activate` every subsystem in `topoSort` order.

The **global barrier between the sweeps is the whole point**. Per-subsystem
Prepare-then-Activate is just `Start` renamed and buys nothing. With two sweeps,
"no subsystem serves until every subsystem has acquired" is guaranteed by the
orchestrator's structure and does not depend on any `DependsOn` edge being
remembered — which is exactly D-11's stated rationale. (The specific
audit-projection ordering is fixed separately by 07-10's edge; D-11 and D-14 are
complementary, not redundant.)

**2. Three methods, not four: `Prepare` / `Activate` / `Stop`.**
Do **not** invent `Unprepare` and `Deactivate`. `Stop` becomes the single
teardown for both states, contracted to be safe on a subsystem that was prepared
but never activated. `Stop`'s existing "MUST be idempotent and MUST NOT block
indefinitely" contract (`subsystem.go:55-57`) already covers this; extend the doc
to name the prepared-only case explicitly.

**3. Rollback:**
- `Prepare` fails at index i → `Stop` the i already-prepared subsystems in
  reverse order. Nothing activated, so no separate deactivate path.
- `Activate` fails at index j → `Stop` **everything prepared** in reverse order.
  The prepared set is a superset of the activated set, so one teardown walk
  covers both. This is why three methods suffice.
- The orchestrator tracks `preparedOrder` and `activatedOrder`. `StopAll` walks
  `preparedOrder` in reverse (it subsumes `activatedOrder`) and honors the
  deadline 07-10 gave it.
- First-failure abort is preserved (`orchestrator.go:58-65`): a non-nil error
  from either phase is fatal — the server does not start.

### D-13.2 — the `Start`-MUST-be-idempotent contract. SETTLED: **keep it, per-subsystem, with the correct reason.**

CONTEXT.md says the contract *"exists to support the pre-start hack"* and may be
vestigial once eager starts die. **RESEARCH found no second reason. PATTERNS
found one, and it is real.** Verified live at
`internal/access/setup/subsystem.go:69-78`:

> `// Start is idempotent: if the subsystem is already started, it returns nil`
> `// immediately. This allows the ABAC subsystem to be pre-started in core`
> `// boot when admin handler construction needs Resolver() before the`
> `// orchestrator drives StartAll. Mirrors store.DatabaseSubsystem.Start.`
> ```
> if s.stack != nil {
>     return nil // already started — guard against double-start (would launch a duplicate poller goroutine)
> }
> ```

The parenthetical — *"would launch a duplicate poller goroutine"* — is a
**second, independent reason** that survives D-09. The guard is **not purely** a
pre-start artifact.

**`cluster.registry` is a second confirmed instance, found in the corrected
census.** `Start` guards at `internal/cluster/heartbeat.go:26`:

> `if r.subAlive != nil { return nil // already started }`

Its side effects are unambiguously non-idempotent: four NATS subscriptions
(`:30-64`), a heartbeat publish (`:72`), and **two goroutines** —
`runHeartbeatTicker` (`:82`) and `runEvictionSweeper` (`:87`), both fenced by
`r.wg`. A second `Start` without the guard would double-subscribe and leak two
goroutines that `Stop` (`:110`) could never fence, because `Stop` nils the
channel/ticker fields it uses to signal them. **Verdict: KEEP**, and the true
reason is a real side effect, not a pre-start artifact. Its guard also carries
**no** pre-start rationale text today, so there is nothing to delete there —
record the verdict, change nothing but the method name it lives under.

Note the guard's shape: it keys on `r.subAlive != nil`, which after the split is
an **Activate**-owned field. Whichever phase the subscriptions land in owns this
guard; do not leave it stranded on a `Prepare` that no longer sets `subAlive`.

**Therefore: keep the MUST, and fix the reason.** The interface contract becomes:
`Prepare` and `Activate` MUST each be idempotent, because a subsystem may guard a
non-idempotent side effect (launching a goroutine, binding a listener) and the
orchestrator does not promise to call each phase exactly once under rollback. The
**false** half of the rationale — "so the subsystem can be pre-started in core
boot before the orchestrator drives StartAll" — dies with 07-09's eager starts
and MUST be deleted wherever it appears. This is decided **per-subsystem**, not
by blanket retirement: audit each guard, keep the ones protecting a real side
effect, and delete only the pre-start justification text.

### ⚠️ Rollback is UNSAFE as originally drafted — two independent bugs (cross-AI review, HIGH)

Both verified live against `internal/lifecycle/orchestrator.go:44-78`. Fix both;
each has its own test in `<behavior>`.

**Bug 1 — the failing subsystem is never cleaned up.** Today `StartAll` appends
to `startOrder` **only after** `sub.Start(ctx)` succeeds (`:56-67`), and rollback
walks only `startOrder`. So the subsystem that **just failed** is never stopped —
even though a failed `Prepare` is precisely the case most likely to have
**partially acquired** resources (half its connections dialed, a listener bound,
a file handle open) before erroring. The naive Prepare loop inherits this exactly.
→ **Append the id to `preparedOrder` BEFORE calling `Prepare`, not after** (or
otherwise ensure the failing target is included in the rollback walk). This makes
`Stop` responsible for tolerating a subsystem that never fully prepared — which is
already the contract this task writes into `Stop`'s doc: *"MUST be safe to call on
a subsystem that was prepared but never activated; MUST be idempotent."* Extend
that sentence to cover **partially prepared** explicitly. `StopAll` walking
`preparedOrder` in reverse then cleans the failing target first, which is correct
ordering.

**Bug 2 — rollback inherits the cancelled startup context.** Rollback currently
calls `o.StopAll(ctx)` with the **startup ctx** (`:58`). After 07-10 makes
`StopAll` deadline-aware, this becomes actively harmful: if the startup ctx is
what *caused* the failure (SIGINT during boot, a boot deadline elapsing), it is
**already cancelled**, so a deadline-aware `StopAll` sees `ctx.Done()` closed and
may **abandon every Stop immediately** — leaking exactly the resources rollback
exists to release. 07-10 made `StopAll` correct-vs-hang; this makes it
correct-vs-cancel.
→ **Roll back on a fresh, bounded context**, never the startup ctx:
`stopCtx, cancel := context.WithTimeout(context.Background(), <bound>); defer cancel()`
then `o.StopAll(stopCtx)`. Use `context.Background()` as the parent — deriving
from the startup ctx re-inherits the cancellation. Reuse 07-10's existing
shutdown bound rather than inventing a new constant; `core.go:982-989`'s
5s-`context.Background()` stop is the in-repo precedent for this exact shape.
⚠️ `StopAll`'s **normal** (non-rollback) callers keep passing their own ctx —
this fresh-context rule is for the **rollback path inside `StartAll`** only.
