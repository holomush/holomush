### D-13.0 ‚Äî what "acquire" and "serve" MEAN. SETTLED. **This supersedes the earlier universal-barrier framing, which was INCOHERENT (cross-AI review, HIGH; verified).**

An earlier revision asserted a universal barrier ‚Äî *"nothing serves until
everything is acquired"* ‚Äî and then required `eventbus` (`go s.server.Start()`)
and `audit` (projection + consumers + retention worker) to move to `Activate`.
**That pair cannot both hold, and the plan could not have booted.** Verified live
in this worktree:

- `go s.server.Start()` (`internal/eventbus/subsystem.go:164`) lives **inside**
  `connectEmbedded()`, which immediately calls `ReadyForConnections()` (`:166`),
  `nats.Connect(nats.InProcessServer(s.server))` (`:177-185`) and
  `jetstream.New(conn)` (`:193`). `connect()` is called from `Start` at `:91`.
  **Server-start and client-connect are one atomic call.** You cannot move
  `go s.server.Start()` into `Activate` while `s.connect()` stays in `Prepare` ‚Äî
  there is no seam between them.
- `audit.Subsystem.Start` (`internal/eventbus/audit/subsystem.go:273-279`) reads
  `s.jsProv.JS()` and **fails closed** with `AUDIT_DEP_NOT_STARTED` when it is
  nil; `projection.go:108` makes a live `js.CreateOrUpdateConsumer` **network
  call**. Audit's acquisition therefore REQUIRES a running bus.

So embedded NATS *running* is the substrate audit must *acquire against*. Under a
literal universal barrier, audit's acquisition would require something the
barrier forbids until acquisition completes. The barrier as stated was circular.

**The error was the definition of "serving", not the design.** Fixed by naming it
precisely. The decisive fact is one line: `internal/eventbus/subsystem.go:153`
sets **`DontListen: true`** on the embedded server's options. The embedded NATS
server **binds no socket and accepts no external connection** ‚Äî it is reachable
only in-process via `nats.InProcessServer(s.server)` (`:179`). Calling it
"serving" was simply wrong: it is process-internal substrate, exactly like a
`pgxpool` handle.

**The definitions, settled ‚Äî use these verbatim in the interface doc comments:**

| Phase | Contract |
|---|---|
| **`Prepare`** | Acquire and wire everything the subsystem needs, **including bringing up process-internal substrate that other subsystems must acquire against** (the embedded NATS server, the DB pool, JetStream streams/durable consumers, plugin processes). MUST NOT expose any **domain** surface reachable from outside the process, and MUST NOT start a work loop that carries **domain traffic**. **Observability surfaces are excepted** (exception 1 below): the NATS Prometheus monitor binds an operator-facing HTTP port here, which is why this clause says "domain" and not "any". MUST be idempotent. |
| **`Activate`** | Begin serving: bind/accept **externally reachable** connections, and start work loops that **carry domain traffic** (deliver events, dispatch commands, sweep, publish). MUST assume every subsystem has already prepared. MUST be idempotent. |
| **`Stop`** | Single teardown for prepared-only, partially-prepared, and activated subsystems. |

**The guarantee this actually buys ‚Äî state it exactly, do not overclaim:**

> **No subsystem accepts externally-reachable DOMAIN traffic, and no domain work
> loop runs, until every subsystem has finished acquiring.**

‚öÝÔ∏è **"Domain" is load-bearing** ‚Äî exception 1 below permits the NATS Prometheus
monitor to bind an operator-facing HTTP port in `Prepare`. An unqualified
"externally-reachable traffic" would make this guarantee contradict that
exception, i.e. the plan would forbid on one line what it permits 25 lines later.
Observability is not domain traffic.

That is strictly weaker than "nothing serves until everything is acquired" and
strictly stronger than today. Crucially, **it is exactly the guarantee D-11's
motivating bug needs**: `grpcSubsystem.DependsOn()` omitted `AuditProjection`, so
gRPC could accept external RPCs before the audit projection was up. Under this
barrier gRPC's listener cannot bind until audit has prepared ‚Äî **regardless of any
`DependsOn` edge**. The class is still unrepresentable.

**Why ordering inside a sweep still saves audit:** Sweep 1 walks `topoSort` order,
so `Prepare(eventbus)` runs before `Prepare(audit)` via audit's existing
`DependsOn(Database, EventBus, Plugins)` (`audit/subsystem.go:255-261`). Intra-sweep
`DependsOn` ordering is unchanged and is what acquisition depends on; the barrier
governs only the **sweep boundary**. Both mechanisms are live and complementary.

**Two deliberate exceptions, recorded rather than hidden:**
1. **The NATS Prometheus monitor / exporter** (`subsystem.go:109-124`;
   `HTTPPort: s.cfg.MonitorPort` at `:156`) DOES bind an operator-facing HTTP
   port during `Prepare`. It is an **observability** surface, not a domain one,
   and it is the same class as the observability server that already starts
   before `StartAll` today (`core.go:351`). Keep it in `Prepare` with a comment
   naming this exception; do not contort the split around it.
2. **External-mode eventbus** (`ModeExternal`, `connectExternal`) dials a remote
   cluster in `Prepare`. Dialling outbound is acquisition, not serving. No change.

### D-13.1 ‚Äî two-phase rollback semantics. SETTLED. **There is no in-tree analog; this is genuine design work.**

`Orchestrator` has exactly one phase today; `StopAll` walks only `startOrder`
(`orchestrator.go:81-96`). "Activate fails after N prepared" has no precedent
anywhere in the repo. The design:

**1. `StartAll` = two full sweeps, not per-subsystem Prepare-then-Activate.**
- Sweep 1: `Prepare` every subsystem in `topoSort` order.
- Sweep 2: `Activate` every subsystem in `topoSort` order.

The **global barrier between the sweeps is the whole point**. Per-subsystem
Prepare-then-Activate is just `Start` renamed and buys nothing. With two sweeps,
D-13.0's guarantee ‚Äî *no subsystem accepts externally-reachable DOMAIN traffic
and no domain work loop runs until every subsystem has acquired* ‚Äî is enforced by the
orchestrator's structure and does not depend on any `DependsOn` edge being
remembered, which is exactly D-11's stated rationale. (The specific
audit-projection ordering is fixed separately by 07-10's edge; D-11 and D-14 are
complementary, not redundant.)

‚öÝÔ∏è **`DependsOn` ordering is NOT retired ‚Äî it is what makes acquisition work.**
Sweep 1 walks `topoSort` order, so a subsystem whose Prepare acquires against
another's prepared substrate (audit ‚Üí eventbus's JetStream; world ‚Üí the DB pool)
still gets the ordering it needs *within* the sweep. The barrier governs the
**sweep boundary**, not intra-sweep order. Read D-13.0 before touching the split.

**2. Three methods, not four: `Prepare` / `Activate` / `Stop`.**
Do **not** invent `Unprepare` and `Deactivate`. `Stop` becomes the single
teardown for both states, contracted to be safe on a subsystem that was prepared
but never activated. `Stop`'s existing "MUST be idempotent and MUST NOT block
indefinitely" contract (`subsystem.go:55-57`) already covers this; extend the doc
to name the prepared-only case explicitly.

**3. Rollback:**
- `Prepare` fails at index i ‚Üí `Stop` the i already-prepared subsystems in
  reverse order. Nothing activated, so no separate deactivate path.
- `Activate` fails at index j ‚Üí `Stop` **everything prepared** in reverse order.
  The prepared set is a superset of the activated set, so one teardown walk
  covers both. This is why three methods suffice.
- The orchestrator tracks `preparedOrder` and `activatedOrder`. `StopAll` walks
  `preparedOrder` in reverse (it subsumes `activatedOrder`) and honors the
  deadline 07-10 gave it.
- First-failure abort is preserved (`orchestrator.go:58-65`): a non-nil error
  from either phase is fatal ‚Äî the server does not start.

### D-13.2 ‚Äî the `Start`-MUST-be-idempotent contract. SETTLED: **keep it, per-subsystem, with the correct reason.**

CONTEXT.md says the contract *"exists to support the pre-start hack"* and may be
vestigial once eager starts die. **RESEARCH found no second reason. PATTERNS
found one, and it is real.** Verified live at
`internal/access/setup/subsystem.go:69-78`:

> `// Start is idempotent: if the subsystem is already started, it returns nil`
> `// immediately. This allows the ABAC subsystem to be pre-started in core`
> `// boot when admin handler construction needs Resolver() before the`
> `// orchestrator drives StartAll. Mirrors store.DatabaseSubsystem.Start.`
> ```
> if s.stack != nil {
>     return nil // already started ‚Äî guard against double-start (would launch a duplicate poller goroutine)
> }
> ```

The parenthetical ‚Äî *"would launch a duplicate poller goroutine"* ‚Äî is a
**second, independent reason** that survives D-09. The guard is **not purely** a
pre-start artifact.

**`cluster.registry` is a second confirmed instance, found in the corrected
census.** `Start` guards at `internal/cluster/heartbeat.go:26`:

> `if r.subAlive != nil { return nil // already started }`

Its side effects are unambiguously non-idempotent: four NATS subscriptions
(`:30-64`), a heartbeat publish (`:72`), and **two goroutines** ‚Äî
`runHeartbeatTicker` (`:82`) and `runEvictionSweeper` (`:87`), both fenced by
`r.wg`. A second `Start` without the guard would double-subscribe and leak two
goroutines that `Stop` (`:110`) could never fence, because `Stop` nils the
channel/ticker fields it uses to signal them. **Verdict: KEEP**, and the true
reason is a real side effect, not a pre-start artifact. Its guard also carries
**no** pre-start rationale text today, so there is nothing to delete there ‚Äî
record the verdict, change nothing but the method name it lives under.

Note the guard's shape: it keys on `r.subAlive != nil`, which after the split is
an **Activate**-owned field. Whichever phase the subscriptions land in owns this
guard; do not leave it stranded on a `Prepare` that no longer sets `subAlive`.

**Therefore: keep the MUST, and fix the reason.** The interface contract becomes:
`Prepare` and `Activate` MUST each be idempotent, because a subsystem may guard a
non-idempotent side effect (launching a goroutine, binding a listener) and the
orchestrator does not promise to call each phase exactly once under rollback. The
**false** half of the rationale ‚Äî "so the subsystem can be pre-started in core
boot before the orchestrator drives StartAll" ‚Äî dies with 07-09's eager starts
and MUST be deleted wherever it appears. This is decided **per-subsystem**, not
by blanket retirement: audit each guard, keep the ones protecting a real side
effect, and delete only the pre-start justification text.
