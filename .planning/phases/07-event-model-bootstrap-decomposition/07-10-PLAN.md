### LOW-7 ŌĆö SETTLED: `StopAll` takes the deadline AND honors it

`cmd/holomush/core.go:1105` is `defer orch.StopAll(context.Background())` ŌĆö **no
deadline**. A hung plugin subprocess or NATS drain means the process never
self-exits (mitigated only externally by systemd/k8s). `StopAll`
(`orchestrator.go:81-96`) returns nothing and never checks `ctx.Done()`.

Give `StopAll` a 5s deadline **and** make it honor `ctx.Done()`. Passing the ctx
alone is insufficient: `Stop` implementations that ignore it would still hang the
loop.

ŌÜĀ’ĖÅ **The deadline context MUST be built inside a deferred closure, not passed as a
deferred argument** ŌĆö Go evaluates deferred arguments at registration, so a
`WithTimeout` written at the defer site starts its timer at boot and is already
dead by shutdown, causing the deadline-aware `StopAll` to abandon every stop. The
in-repo precedent is `cmd/holomush/core.go:255-261` (telemetry) and `:356-362`
(observability). See Task 1's action for the full reasoning and the retraction of
the earlier `defer orch.StopAll(shutdownCtx)` instruction.

ŌÜĀ’ĖÅ `StopAll` is also called by `StartAll`'s rollback path
(`orchestrator.go:60`), so any signature change ripples there. Keep
`StopAll(ctx)`'s signature; change only its body.

ŌÜĀ’ĖÅ **The rollback path MUST move to a fresh bounded context IN THIS PLAN, not in
07-11 (cross-AI round 4, MEDIUM ŌĆö verified).** Today `StartAll`'s rollback
passes the **startup ctx** into `StopAll` (`orchestrator.go:60`). Once this plan
makes `StopAll` deadline-aware, that inheritance becomes actively harmful: when
the startup ctx is what *caused* the failure (SIGINT during boot, a boot
deadline elapsing), it is already cancelled ŌĆö the new `ctx.Done()` check would
**abandon every rollback Stop immediately**, leaking exactly the resources
rollback exists to release. Rev 6 deferred this repair to 07-11 (Wave B), but
D-12 allows Wave B to be deferred, so 07-10 shipping alone would AMPLIFY
abandon-on-cancel for however long Wave B waits. The fix is three lines and
lands in Task 1: rollback builds
`stopCtx, cancel := context.WithTimeout(context.Background(), 5*time.Second)`
(the same in-repo idiom as the shutdown closure) and calls `o.StopAll(stopCtx)`.
Normal (non-rollback) callers keep passing their own ctx. 07-11's two-sweep
rewrite then PRESERVES this ŌĆö it does not introduce it.

### ŌÜĀ’ĖÅ Wire the deadline as a CLOSURE ŌĆö the direct-arg defer REGRESSES shutdown (cross-AI review, HIGH; verified)

An earlier revision prescribed `defer orch.StopAll(shutdownCtx)` and pinned it
with `rg -n 'defer orch.StopAll\(shutdownCtx\)'`. **That is retracted: it is a net
downgrade on today's behavior, and the grep actively forced the broken construct.**

**Go evaluates deferred call arguments at REGISTRATION time, not at execution
time.** A `context.WithTimeout(context.Background(), 5*time.Second)` built at the
defer site therefore starts its timer **at boot**. It expires ~5s into uptime. At
actual shutdown ŌĆö minutes, hours or weeks later ŌĆö `StopAll` receives an
**already-cancelled** context, and Task 1's brand-new `ctx.Err()` check
short-circuits the loop on the **first** subsystem: every graceful stop is
abandoned instantly. Today's `defer orch.StopAll(context.Background())`
(`core.go:1105`) has no deadline but does stop cleanly, so the "fix" would be
strictly worse than the bug.

**The correct pattern already exists in this file** ŌĆö `cmd/holomush/core.go:255-261`
(telemetry shutdown), with a second instance at `:356-362` (observability server).
Both build the timeout **inside** the deferred function:

- register `defer func() { ŌĆ” }()`;
- **inside** the closure, construct
  `shutdownCtx, cancel := context.WithTimeout(context.Background(), 5*time.Second)`;
- `defer cancel()` inside the closure;
- call `orch.StopAll(shutdownCtx)` and log any error with `slog.WarnContext(shutdownCtx, ŌĆ”)`.

Replace `core.go:1105`'s `defer orch.StopAll(context.Background())` with exactly
that closure shape, parented on `context.Background()` (never on the boot ctx ŌĆö a
SIGINT-cancelled boot ctx re-inherits the cancellation this closure exists to
avoid). The timer then starts when the closure runs, which is what "the timeout
must start at shutdown" actually requires.
