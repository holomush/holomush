#### Why the verifier genuinely needs the bus — the edge direction is not a choice

The verifier's `DependsOn(EventBus)` is **not** an artifact of 07-09's rule being
over-broad. It is real, and it is forced by data flow. Verified live:

- `chain.VerifierSubsystemConfig` takes `Repo` + `Handlers`
  (`cmd/holomush/core.go:1053-1057`).
- `Handlers` is `chainHandlers` (`core.go:1047-1052`) =
  `[policy.PolicySetHandlerFor(gameID), dek.RekeyHandlerFor(gameID)]`, plus
  `readStreamW.AuditChainHandler` **iff `readStreamW.Handler != nil`**.
- `readStreamW` comes from `buildReadStreamWiring`, which **hard-requires**
  `AuditPublisher` (`cmd/holomush/readstream_wiring.go:118` — a nil
  `AuditPublisher` returns an empty wiring), and `auditPublisher` is
  `eventbus.NewRenderingPublisher(eventBusSub.Publisher(), verbRegistry)`
  (`core.go:735`).

⇒ **Building the verifier's handler set reads `eventBusSub.Publisher()`.** The bus
must be up before the verifier can know its own chain set.

**So `EventBus → Verifier` is not merely a cycle — it is a silent security
regression.** With that edge, the verifier resolves its `HandlersProvider` *before*
the bus exists → `Publisher()` nil → `buildReadStreamWiring` returns an empty
wiring → `readStreamW.Handler == nil` → **`operator_read` is never registered on
the boot gate**. A broken `operator_read` chain would stop refusing the boot,
silently, while INV-CRYPTO-102 appears satisfied. Rev 4's edge would have made the
gate weaker, not stronger.

The reviewer's suggested fix was to split `cryptoWiring` into (a) verifier
boot-gate inputs not needing EventBus and (b) runtime/admin wiring that may. That
split **is** mechanically available: all three chain handlers are **pure functions

(`readstream_wiring.go:133`) touches no pool and no publisher. Only the
*registration predicate* (`readStreamW.Handler != nil`) couples the verifier to
the bus.

**Rejected, deliberately:** decoupling requires changing **when a fail-closed
security gate registers a chain** — replacing "register `operator_read` iff

availability). That is a **behavior change to an INV-CRYPTO-102 boot gate**, in a
phase whose own boundary is *"three **behavior-preserving** structural changes"*
(07-CONTEXT.md § Phase Boundary). It is not equivalent: today a `NewHandler` error

(`readstream_wiring.go:158-191`), and the inversion would register anyway. The
current conditional is a compromise the code **already documents** as such

own issue, not a side effect of a comment fix. **Left exactly as-is.**

### LOW-7 — SETTLED: `StopAll` takes the deadline AND honors it

`cmd/holomush/core.go:1105` is `defer orch.StopAll(context.Background())` — **no
deadline**. A hung plugin subprocess or NATS drain means the process never
self-exits (mitigated only externally by systemd/k8s). `StopAll`
(`orchestrator.go:81-96`) returns nothing and never checks `ctx.Done()`.

Give `StopAll` a 5s deadline **and** make it honor `ctx.Done()`. Passing the ctx
alone is insufficient: `Stop` implementations that ignore it would still hang the
loop.

⚠️ **The deadline context MUST be built inside a deferred closure, not passed as a
deferred argument** — Go evaluates deferred arguments at registration, so a
`WithTimeout` written at the defer site starts its timer at boot and is already
dead by shutdown, causing the deadline-aware `StopAll` to abandon every stop. The
in-repo precedent is `cmd/holomush/core.go:255-261` (telemetry) and `:356-362`
(observability). See Task 1's action for the full reasoning and the retraction of
the earlier `defer orch.StopAll(shutdownCtx)` instruction.

⚠️ `StopAll` is also called by `StartAll`'s rollback path
(`orchestrator.go:60`), so any signature change ripples there. Keep
`StopAll(ctx)`'s signature; change only its body.

⚠️ **The rollback path MUST move to a fresh bounded context IN THIS PLAN, not in
07-11 (cross-AI round 4, MEDIUM — verified).** Today `StartAll`'s rollback
passes the **startup ctx** into `StopAll` (`orchestrator.go:60`). Once this plan
makes `StopAll` deadline-aware, that inheritance becomes actively harmful: when
the startup ctx is what *caused* the failure (SIGINT during boot, a boot
deadline elapsing), it is already cancelled — the new `ctx.Done()` check would
**abandon every rollback Stop immediately**, leaking exactly the resources
rollback exists to release. Rev 6 deferred this repair to 07-11 (Wave B), but
D-12 allows Wave B to be deferred, so 07-10 shipping alone would AMPLIFY
abandon-on-cancel for however long Wave B waits. The fix is three lines and
lands in Task 1: rollback builds
`stopCtx, cancel := context.WithTimeout(context.Background(), 5*time.Second)`
(the same in-repo idiom as the shutdown closure) and calls `o.StopAll(stopCtx)`.
Normal (non-rollback) callers keep passing their own ctx. 07-11's two-sweep
rewrite then PRESERVES this — it does not introduce it.

### ⚠️ Wire the deadline as a CLOSURE — the direct-arg defer REGRESSES shutdown (cross-AI review, HIGH; verified)

An earlier revision prescribed `defer orch.StopAll(shutdownCtx)` and pinned it
with `rg -n 'defer orch.StopAll\(shutdownCtx\)'`. **That is retracted: it is a net
downgrade on today's behavior, and the grep actively forced the broken construct.**

**Go evaluates deferred call arguments at REGISTRATION time, not at execution
time.** A `context.WithTimeout(context.Background(), 5*time.Second)` built at the
defer site therefore starts its timer **at boot**. It expires ~5s into uptime. At
actual shutdown — minutes, hours or weeks later — `StopAll` receives an
**already-cancelled** context, and Task 1's brand-new `ctx.Err()` check
short-circuits the loop on the **first** subsystem: every graceful stop is
abandoned instantly. Today's `defer orch.StopAll(context.Background())`
(`core.go:1105`) has no deadline but does stop cleanly, so the "fix" would be
strictly worse than the bug.

**The correct pattern already exists in this file** — `cmd/holomush/core.go:255-261`
(telemetry shutdown), with a second instance at `:356-362` (observability server).
Both build the timeout **inside** the deferred function:

- register `defer func() { … }()`;
- **inside** the closure, construct
  `shutdownCtx, cancel := context.WithTimeout(context.Background(), 5*time.Second)`;
- `defer cancel()` inside the closure;
- call `orch.StopAll(shutdownCtx)` and log any error with `slog.WarnContext(shutdownCtx, …)`.

Replace `core.go:1105`'s `defer orch.StopAll(context.Background())` with exactly
that closure shape, parented on `context.Background()` (never on the boot ctx — a
SIGINT-cancelled boot ctx re-inherits the cancellation this closure exists to
avoid). The timer then starts when the closure runs, which is what "the timeout
must start at shutdown" actually requires.
