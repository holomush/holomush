<objective>
Fix a **live correctness bug** (D-07): plugin history pagination is ULID-ordered
in a system whose own code states ULID order does not match stream order. On a
busy stream, plugin history pages can **skip or repeat** events.

Purpose: D-07 is not tidiness. Verified live, hop by hop:
1. `QueryHistory` returns `eventbus.Event` **with** `Seq`
   (`internal/eventbus/types.go:141-148`).
2. `busEventToCoreEvent` **destroyed** it — `core.Event` had no such field
   (deleted in 07-07, which is what unblocks this plan).
3. `encodeHostEventCursor` is therefore forced to hardcode `Seq: 0`, and
   **says so** (`internal/plugin/hostcap/servers.go:1279-1296`).
4. `busHistoryReaderAdapter.ReplayTail` (`cmd/holomush/sub_grpc.go:916-935`)
   passes only `Subject`/`Direction`/`PageSize`/`NotBefore`/`BeforeID` — no seq.
5. `internal/eventbus/history/hot_jetstream.go:424-431`: *"Ordering is owned by
   JetStream per-stream sequence, not ULID lex order — concurrent publishers
   produce events whose ULIDs do NOT match stream sequence."*

**This IS a behavior change** (pagination becomes seq-correct) and MUST ship with
a regression test pinning the failure — see `<red_framing_correction>`, which
**corrects this plan's original RED framing**.

Output: a genuine RED→GREEN seq fix, plus a guard that keeps `Seq` off the
plugin wire (D-08).
</objective>

<red_framing_correction>
### ⚠️ The bug is WORSE than this plan originally diagnosed — and the RED is simpler

This plan (and `07-VALIDATION.md`) previously asserted: *"A quiet-stream page walk
passes today and proves nothing"*, so the RED **required** concurrent publishers
whose ULIDs disagree with stream sequence. **That is FALSE.** An external reviewer
challenged it; it was re-verified against live source in this worktree and the
challenge is correct. Do NOT reinstate the old framing.

**The mechanism, in the codebase's own words** (`internal/eventbus/bus.go:88-105`):

> *"Pagination ordering is by JetStream stream sequence (js_seq), not by ULID.
> Cursors are **(seq, id) pairs** … The id field is a **tripwire** that validates
> the cursor's seq still names the same event in storage … **Zero seq means "from
> the start" (forward) or "from the end" (backward).**"*
> `BeforeID  ulid.ULID // tripwire for BeforeSeq; zero = skip validation`

`BeforeID` is **not a filter**. It is a *validator for a seq that must also be
supplied*. So with `BeforeSeq == 0` — which is exactly what
`busHistoryReaderAdapter.ReplayTail` sends today (`sub_grpc.go:930-936` sets
`q.BeforeID` and never `q.BeforeSeq`) — the query means **"from the end"**, i.e.
the newest page, on **every** call. Confirmed on both tiers:

- **Hot:** `matchesQuery` (`hot_jetstream.go:392-402`) filters on
  `AfterSeq`/`BeforeSeq`/`NotBefore`/`NotAfter` and has **no `BeforeID` branch at
  all** — a lone `BeforeID` excludes nothing. `buildConfig` only takes the cursor
  path `if q.BeforeSeq > 0` (`:338`); otherwise it falls through to *"BACKWARD
  without cursor: tail-oriented read"* (`:348-350`) and returns the newest window.
- **Cold:** `cold_postgres.go:132-134` computes `hasCursor := cursorSeq > 0`, so a
  lone `BeforeID` produces no SQL cursor bound either.

**Consequence: plugin multipage pagination REPEATS the newest page forever, on a
QUIET stream, with no concurrency involved.** That is a deterministic,
easier-to-trigger, and more severe bug than the concurrent skip/repeat originally
described — the original framing understated it.

**Therefore:**
- The RED gate is **Spec A: a quiet-stream multipage walk** (pageSize ≪ total). It
  fails today, deterministically, with no race.
- The **concurrent-publisher case is defence-in-depth, kept post-green** (Spec B):
  it is what proves the cursor advances by *seq* rather than *ID* after the fix.
- Do **not** reject a multipage RED as "tautological". It is the primary proof.

**Doc-drift spotted while verifying (non-blocking, fix opportunistically):**
`internal/eventbus/history/tier.go:719-720` claims *"The ID cursor (AfterID /
BeforeID) is preserved so matchesQuery can still filter at the boundary."*
`matchesQuery` does no such filtering. If Task 2 touches this file, correct the
comment; otherwise note it in the SUMMARY. Do not let it mislead the fix.
</red_framing_correction>

<legacy_zero_seq_policy>
### ⚠️ The promised `Seq == 0` → `BeforeID` fallback DOES NOT EXIST. Policy settled: **restart from the tail.** (cross-AI review, HIGH; verified)

An earlier revision instructed: *"Keep the cold-tier `Seq == 0` ID-only fallback
working — `beforeSeq == 0` MUST mean 'no seq cursor — fall back to `BeforeID`',
**exactly as today**"*, and a threat row (T-07-42) asserted the same. **Retracted:
that fallback is fiction. There is nothing to keep working.** Verified live, on
both tiers:

- `internal/eventbus/bus.go:87,94` — the contract in the codebase's own words:
  *"**Zero seq means 'from the start' (forward) or 'from the end' (backward)**"*
  and `BeforeID` is *"tripwire for BeforeSeq; zero = skip validation"*. `BeforeID`
  **validates a nonzero seq**; it is not a standalone anchor.
- **Hot:** `matchesQuery` (`history/hot_jetstream.go:392-402`) has **no `BeforeID`
  branch** — a lone `BeforeID` excludes nothing. `buildConfig` (`:334,338`) takes
  the cursor path only `if q.BeforeSeq > 0`, else falls through to the
  tail-oriented read.
- **Cold:** `history/cold_postgres.go:125` declares a cursor only when the seq is
  **positive** (`hasCursor := cursorSeq > 0`).

So a legacy zero-seq token **cannot paginate by ID** without a new ID→seq lookup
that does not exist. Describing today's behavior as an "ID fallback" was exactly
backwards: **zero seq is what causes the repeat bug this plan fixes.**

**Policy, settled — restart from the tail (the status quo), explicitly:**

`beforeSeq == 0` means **"no cursor: read the newest page"**. That is what the
code already does and what every tier already agrees on. Concretely:

- `busHistoryReaderAdapter.ReplayTail` sets `q.BeforeSeq` **only** when
  `beforeSeq != 0` (leaving it zero = tail read), and keeps setting `q.BeforeID`
  only when `!beforeID.IsZero()`.
- Do **not** invent an ID→seq resolution path. Do **not** treat 0 as a real
  sequence. Do **not** reject a zero-seq token as stale.

**Why this is safe rather than a silent data-loss window** — the two ways a
zero-seq cursor can arrive, both benign:
1. **A first-page request.** No cursor supplied → tail read. Correct and
   unchanged.
2. **A cursor minted before this change** (every cursor in flight today carries
   `Seq: 0`, because both encoders hardcode it). Such a token already means
   "newest page" to every tier — a plugin holding one is **already** getting the
   repeat. Post-fix it gets the newest page once and then paginates correctly from
   the seq-bearing cursor the fix mints. The cursor is an in-memory pagination
   token, not durable state: it is minted per page and never persisted, so the
   legacy population drains within one page-walk.

**Rejected alternatives, recorded:**
- *Reject-as-stale* (`ErrCursorStale`): would break case 1 — a legitimate
  first-page request also carries `beforeSeq == 0` and is indistinguishable from
  a legacy token. It would turn every plugin's first history call into an error.
- *ID→seq resolution*: real new machinery (a lookup by ULID across both tiers) to
  serve a token population that drains in one page. Not justified. If a future
  need arises it is an explicit design, not a silent addition here.

⚠️ Do NOT reinstate the "ID fallback" language anywhere in this plan, its tests,
or its SUMMARY. A test named for an "ID-only fallback" would pin a behavior the
system does not have.
</legacy_zero_seq_policy>

<page_advance_anchor>
### ⚠️ The page-advance anchor is the **OLDEST** event (index 0), NOT the last. (cross-AI review, HIGH; verified live)

An earlier revision instructed *"encode from the **last** event of each page"* and
*"pass the **last** event's real `e.Seq`"* — while citing `hostcap/servers.go` as
the precedent. **It misdescribed the very file it cited.** Retracted. Verified
live in this worktree:

- `internal/plugin/hostcap/servers.go:900-906`, in the codebase's own words:
  > *"Populate next_cursor from the **oldest (first)** event in the page, which is
  > the pagination anchor for the next backward read. ReplayTail returns events in
  > ascending order (oldest→newest), so **index 0** is the boundary."*
  > `nextCursor = protoEvents[0].GetCursor()`
- `internal/plugin/hostfunc/stdlib_focus.go:452-462` — the Lua path agrees:
  `Host: &cursor.HostCursor{Seq: 0, ID: events[0].ID}`, with the comment *"The
  oldest event (index 0, ascending order) is the backward-pagination anchor."*
- `cmd/holomush/sub_grpc.go:913-915` confirms the ascending order the above relies on.

**Why "last" is not a cosmetic slip — it silently reintroduces the bug this plan
exists to fix.** `BeforeSeq` is an **exclusive upper bound**. On a **backward**
walk the next page must start below the **oldest** event already returned. Feeding
the **newest** event's seq back as `BeforeSeq` re-requests almost the entire page
just returned — a near-full page repeat, i.e. D-07's failure mode arriving by a
different mechanism.

⚠️ **It would pass every acceptance criterion in an earlier revision**, because
they assert a **real** (nonzero) `Seq` arrives — not that the **right** event's
`Seq` arrives. Spec A is the only thing that catches it, and only because it
asserts no ID appears in two pages.

**Two DIFFERENT things, do not conflate them:**

| Site | Which event's `Seq` | Why |
|---|---|---|
| **Per-event** cursor (`servers.go:896`; `stdlib_focus.go:437-441`) | **each event's own `e.Seq`** | every event carries its own cursor |
| **`next_cursor`** / page-advance anchor (`servers.go:904`; `stdlib_focus.go:459-463`) | the **oldest** event's — `protoEvents[0]` / `events[0]` | it is the exclusive upper bound for the next backward page |

**Mechanical consequence worth internalizing:** once the per-event encode carries
the real `Seq`, `next_cursor` needs **no separate anchor logic at all** — it
simply reuses `protoEvents[0]`'s already-correct cursor, exactly as
`servers.go:904` already does. The existing `[0]` indexing is **already right**;
only the hardcoded `Seq: 0` inside the cursor is wrong. Do not "fix" the index.
</page_advance_anchor>

<do_not_reintroduce>
⚠️ **D-08 — do NOT expose `Seq` to plugins.** `hostv1.Event`
(`api/proto/holomush/plugin/host/v1/stream.proto:45-71`) has **8 fields and no
seq** — verified live. The `cursor` field (`= 8`) is documented there as an
**opaque** blob, encoded by `internal/eventbus/cursor`, an `internal/` package an
external plugin cannot import. Plugins therefore see **no seq today**, hardcoded
or otherwise, so D-07 is a **zero-contract-change** fix.

Adding a plugin-facing `seq` would (a) violate `internal/eventbus/types.go:143`
(*"Seq … Host-internal — never serialized in any public proto envelope"*), and
(b) hand plugins a number whose meaning silently changes across tiers —
`history/tier.go:713-718`: a cold event's `js_seq` is an aged-out JS seq no
longer in JS retention; seq is only meaningful within a tier's sequence space.
A future plugin needing independent ordering requires an explicit invariant
amendment + ADR, not a silent override.
</do_not_reintroduce>
