<objective>
Fix a **live correctness bug** (D-07): plugin history pagination is ULID-ordered
in a system whose own code states ULID order does not match stream order. On a
busy stream, plugin history pages can **skip or repeat** events.

Purpose: D-07 is not tidiness. Verified live, hop by hop:
1. `QueryHistory` returns `eventbus.Event` **with** `Seq`
   (`internal/eventbus/types.go:141-148`).
2. `busEventToCoreEvent` **destroyed** it â€” `core.Event` had no such field
   (deleted in 07-07, which is what unblocks this plan).
3. `encodeHostEventCursor` is therefore forced to hardcode `Seq: 0`, and
   **says so** (`internal/plugin/hostcap/servers.go:1279-1296`).
4. `busHistoryReaderAdapter.ReplayTail` (`cmd/holomush/sub_grpc.go:916-935`)
   passes only `Subject`/`Direction`/`PageSize`/`NotBefore`/`BeforeID` â€” no seq.
5. `internal/eventbus/history/hot_jetstream.go:424-431`: *"Ordering is owned by
   JetStream per-stream sequence, not ULID lex order â€” concurrent publishers
   produce events whose ULIDs do NOT match stream sequence."*

**This IS a behavior change** (pagination becomes seq-correct) and MUST ship with
a regression test pinning the failure â€” see `<red_framing_correction>`, which
**corrects this plan's original RED framing**.

Output: a genuine REDâ†’GREEN seq fix, plus a guard that keeps `Seq` off the
plugin wire (D-08).

<red_framing_correction>
### âš ď¸Ź The bug is WORSE than this plan originally diagnosed â€” and the RED is simpler

This plan (and `07-VALIDATION.md`) previously asserted: *"A quiet-stream page walk
passes today and proves nothing"*, so the RED **required** concurrent publishers
whose ULIDs disagree with stream sequence. **That is FALSE.** An external reviewer
challenged it; it was re-verified against live source in this worktree and the
challenge is correct. Do NOT reinstate the old framing.

**The mechanism, in the codebase's own words** (`internal/eventbus/bus.go:88-105`):

> *"Pagination ordering is by JetStream stream sequence (js_seq), not by ULID.
> Cursors are **(seq, id) pairs** â€¦ The id field is a **tripwire** that validates
> the cursor's seq still names the same event in storage â€¦ **Zero seq means "from
> the start" (forward) or "from the end" (backward).**"*
> `BeforeID  ulid.ULID // tripwire for BeforeSeq; zero = skip validation`

`BeforeID` is **not a filter**. It is a *validator for a seq that must also be
supplied*. So with `BeforeSeq == 0` â€” which is exactly what
`busHistoryReaderAdapter.ReplayTail` sends today (`sub_grpc.go:930-936` sets
`q.BeforeID` and never `q.BeforeSeq`) â€” the query means **"from the end"**, i.e.
the newest page, on **every** call. Confirmed on both tiers:

- **Hot:** `matchesQuery` (`hot_jetstream.go:392-402`) filters on
  `AfterSeq`/`BeforeSeq`/`NotBefore`/`NotAfter` and has **no `BeforeID` branch at
  all** â€” a lone `BeforeID` excludes nothing. `buildConfig` only takes the cursor
  path `if q.BeforeSeq > 0` (`:338`); otherwise it falls through to *"BACKWARD
  without cursor: tail-oriented read"* (`:348-350`) and returns the newest window.
- **Cold:** `cold_postgres.go:132-134` computes `hasCursor := cursorSeq > 0`, so a
  lone `BeforeID` produces no SQL cursor bound either.

**Consequence: plugin multipage pagination REPEATS the newest page forever, on a
QUIET stream, with no concurrency involved.** That is a deterministic,
easier-to-trigger, and more severe bug than the concurrent skip/repeat originally
described â€” the original framing understated it.

**Therefore:**
- The RED gate is **Spec A: a quiet-stream multipage walk** (pageSize â‰Ş total). It
  fails today, deterministically, with no race.
- The **concurrent-publisher case is defence-in-depth, kept post-green** (Spec B):
  it is what proves the cursor advances by *seq* rather than *ID* after the fix.
- Do **not** reject a multipage RED as "tautological". It is the primary proof.

**Doc-drift spotted while verifying (non-blocking, fix opportunistically):**
`internal/eventbus/history/tier.go:719-720` claims *"The ID cursor (AfterID /
BeforeID) is preserved so matchesQuery can still filter at the boundary."*
`matchesQuery` does no such filtering. If Task 2 touches this file, correct the
comment; otherwise note it in the SUMMARY. Do not let it mislead the fix.
</red_framing_correction>

<legacy_zero_seq_policy>
### âš ď¸Ź The promised `Seq == 0` â†’ `BeforeID` fallback DOES NOT EXIST. Policy settled: **restart from the tail.** (cross-AI review, HIGH; verified)

An earlier revision instructed: *"Keep the cold-tier `Seq == 0` ID-only fallback
working â€” `beforeSeq == 0` MUST mean 'no seq cursor â€” fall back to `BeforeID`',
**exactly as today**"*, and a threat row (T-07-42) asserted the same. **Retracted:
that fallback is fiction. There is nothing to keep working.** Verified live, on
both tiers:

- `internal/eventbus/bus.go:87,94` â€” the contract in the codebase's own words:
  *"**Zero seq means 'from the start' (forward) or 'from the end' (backward)**"*
  and `BeforeID` is *"tripwire for BeforeSeq; zero = skip validation"*. `BeforeID`
  **validates a nonzero seq**; it is not a standalone anchor.
- **Hot:** `matchesQuery` (`history/hot_jetstream.go:392-402`) has **no `BeforeID`
  branch** â€” a lone `BeforeID` excludes nothing. `buildConfig` (`:334,338`) takes
  the cursor path only `if q.BeforeSeq > 0`, else falls through to the
  tail-oriented read.
- **Cold:** `history/cold_postgres.go:125` declares a cursor only when the seq is
  **positive** (`hasCursor := cursorSeq > 0`).

So a legacy zero-seq token **cannot paginate by ID** without a new IDâ†’seq lookup
that does not exist. Describing today's behavior as an "ID fallback" was exactly
backwards: **zero seq is what causes the repeat bug this plan fixes.**

**Policy, settled â€” restart from the tail (the status quo), explicitly:**

`beforeSeq == 0` means **"no cursor: read the newest page"**. That is what the
code already does and what every tier already agrees on. Concretely:

- `busHistoryReaderAdapter.ReplayTail` sets `q.BeforeSeq` **only** when
  `beforeSeq != 0` (leaving it zero = tail read), and keeps setting `q.BeforeID`
  only when `!beforeID.IsZero()`.
- Do **not** invent an IDâ†’seq resolution path. Do **not** treat 0 as a real
  sequence. Do **not** reject a zero-seq token as stale.

**Why this is safe rather than a silent data-loss window** â€” the two ways a
zero-seq cursor can arrive, both benign:
1. **A first-page request.** No cursor supplied â†’ tail read. Correct and
   unchanged.
2. **A cursor minted before this change** (every cursor in flight today carries
   `Seq: 0`, because both encoders hardcode it). Such a token already means
   "newest page" to every tier â€” a plugin holding one is **already** getting the
   repeat. Post-fix it gets the newest page once and then paginates correctly from
   the seq-bearing cursor the fix mints. The cursor is an in-memory pagination
   token, not durable state: it is minted per page and never persisted, so the
   legacy population drains within one page-walk.

**Rejected alternatives, recorded:**
- *Reject-as-stale* (`ErrCursorStale`): would break case 1 â€” a legitimate
  first-page request also carries `beforeSeq == 0` and is indistinguishable from
  a legacy token. It would turn every plugin's first history call into an error.
- *IDâ†’seq resolution*: real new machinery (a lookup by ULID across both tiers) to
  serve a token population that drains in one page. Not justified. If a future
  need arises it is an explicit design, not a silent addition here.

âš ď¸Ź Do NOT reinstate the "ID fallback" language anywhere in this plan, its tests,
or its SUMMARY. A test named for an "ID-only fallback" would pin a behavior the
system does not have.
</legacy_zero_seq_policy>

amendment + ADR, not a silent override.
