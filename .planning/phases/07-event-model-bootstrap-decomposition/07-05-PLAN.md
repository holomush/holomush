### 3. FINDING-5 (BLOCKING, cross-AI review) — an `eventbus.Publisher` ALONE cannot qualify a subject

Both external reviewers independently found this; verified live in this worktree.
An `Emitter` holding only a `Publisher` **cannot emit at all** — it would fail at
publish, on every arrive/leave/session_ended, on the first boot:

- Qualification obtains the game ID **from the bus, not the publisher**:
  `busEventAppender` holds `bus *eventbus.Subsystem` and does
  `gameID := b.bus.GameID()` → `eventbus.Qualify(gameID, event.Stream)`
  (`cmd/holomush/sub_grpc.go:821-828`). `eventbus.Publisher` is a **one-method
  interface** (`Publish(ctx, Event) error`) — it exposes no game ID.
- `eventbus.Qualify` **rejects** a relative ref with an empty game ID:
  `"game id required to qualify stream reference"` (`qualify.go:29-31`).
- `NewSubject` **rejects** any subject whose first token is not `events`
  (`types.go:257-258`), and the publisher's own `contextIDFromSubject` requires the
  `events.` prefix (`publisher.go:518-521`).

So emitting `"location.<id>"` or `"system"` is a hard failure, not a subtle drift.

**SETTLED: `Emitter` carries a `gameID func() string` alongside the Publisher.**
This is **already the in-repo idiom for exactly this problem** — verified live:
`busHistoryReaderAdapter` (`cmd/holomush/sub_grpc.go:906-909`) holds
`{ reader eventbus.HistoryReader; gameID func() string }` for precisely this
reason, and `cmd/holomush/sub_grpc_adapters_test.go:312` constructs it with
`gameID: func() string { return "main" }`. Copy that shape; do not invent a
`GameIDProvider` interface. **Carry the `gameID == "" → "main"` fallback verbatim**
— both `busEventAppender.Append` (`:822-825`) and
`busHistoryReaderAdapter.ReplayTail` (`:921-924`) have it, and dropping it changes
behavior on an un-initialized game ID.

### 4. FINDING-6 (cross-AI review) — use `eventbus.NewEvent`, and convert the vocabulary type explicitly

- **Do NOT build a raw `eventbus.Event{}` literal.** `eventbus.NewEvent(subject, typ, actor, payload)`
  (`internal/eventbus/types.go:206-215`) is the canonical construction path; its doc
  says callers "MUST still use NewEvent for the base value rather than a raw
  `Event{}` literal", and it stamps the monotonic ULID via `core.NewULID()` for you.
  That also satisfies CLAUDE.md § ULID Generation without a hand-rolled ID.
- **`eventvocab.EventType` is NOT assignable to `eventbus.Type`.** They are distinct
  named types. Today `busEventAppender` converts via
  `eventbus.NewType(string(event.Type))` and handles its error
  (`sub_grpc.go:829-832`). Do the same: `eventbus.NewType(string(eventvocab.EventTypeArrive))`,
  wrapping the error as that site does. Do not paper over it with a bare cast.

### 5. `Engine.EndSession`'s fate — SETTLED: it comes along, named deliberately

`Engine` has **three** methods, not two: `HandleConnect` (`engine.go:65`),
`HandleDisconnect` (`engine.go:88`), and `EndSession`
(`engine_end_session.go:33`). D-04 renames only the first two. `EndSession` has
6 live call sites across `auth`, `grpc`, and `cmd/holomush`.

**SETTLED: all three live on `presence.Emitter`, with the verb set
`EmitArrive` / `EmitLeave` / `EmitSessionEnded`** (D-04 delegates the verb set to
the planner). Rationale: the three share the sink and the marshal→build→publish
shape, so splitting them would produce two types over one Publisher.
`Arrived`/`Departed` (D-04's other candidate) cannot absorb the third without a
mixed-tense set (`Arrived`/`Departed`/`EndSession`); `Emit*` is uniform and each
name states the event type it emits (`arrive`/`leave`/`session_ended`), which is
honest — D-04 itself notes "Engine" is fiction and the type "is an event
emitter".
</must_settle>
