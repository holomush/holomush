<planner_settled_decisions>
**`Close` and `UnloadPlugin` are assigned to the LOAD unit.** Research left this open; here is the
decision and why. Both are inverses of load-unit operations — `Close` inverts `LoadAll`,
`UnloadPlugin` inverts `loadPlugin` — and both need `policyInstaller`, `hosts`, and `luaHost`, which
are load-only collaborators. Putting them in the runtime unit would force the runtime unit to hold
those three fields purely for teardown, inverting the orchestration direction: `loadPlugin` already
drives runtime and identity forward through narrow interfaces, so its inverses should drive them
backward the same way. Assigning teardown to the runtime unit would give us two orchestrators
pointing at each other, which is the coupling this phase exists to remove.

**`m.hosts[TypeLua]` vs `m.luaHost` is NOT collapsed.** Restated as a prohibition above. File a
follow-up GitHub issue at phase close referencing #4674's original item.

**Lock discipline across three units.** `loadPlugin`'s four critical sections are already disjoint
acquisitions. After this plan they are: sections 1 and 2 → runtime unit (inflight claim/release),
sections 3 and 4 → identity unit (register/rollback), section 5 → runtime unit (commit). Each is a
separate short acquisition of that unit's own lock; the loader itself holds no lock across any of
them. This preserves the existing interleaving window exactly and introduces no nested acquisition.
</planner_settled_decisions>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| filesystem → `Discover` | Plugin manifests are parsed from `pluginsDir`; a malformed or hostile manifest crosses here. `Discover` inlines the parse (there is no separate `loadManifest`). |
| manifest → trust allowlist | `trustAllowlist` gates server-side trust escalation; `warnUnknownTrustAllowlistEntries` reports entries that match no plugin. |
| load orchestration → runtime + identity state | `loadPlugin` and `UnloadPlugin` mutate registry state across three units through four disjoint critical sections. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-8-25 | Tampering | `UnloadPlugin`'s cache-cleanup-before-early-return order | **high** | mitigate | Cleanup runs first and unconditionally so a repeat unload is idempotent. Moving it after the early return would leave orphaned `activeByName` / `loaded` / `pluginHosts` entries — state corruption that only appears on the second call and would pass a single-unload test. Controls: an explicit prohibition, verbatim body move, and the four existing `UnloadPlugin` tests passing unedited. |
| T-8-06 | Denial of Service | Lock-ordering deadlock across three unit locks | **high** | mitigate | The loader holds no lock across any of `loadPlugin`'s four sections; each section takes and releases exactly one unit's lock. Controls: an explicit prohibition, a required read-through recorded in the SUMMARY, Task 1's instruction not to add a speculative loader lock, and `-race` under `task test:int` where the lane supports it. |
| T-8-05 | Tampering | TOCTOU between identity and runtime state during load | medium | mitigate | The window between the identity write (section 3) and the runtime commit (section 5) already exists on `main` — the lock is released between them today. The extraction preserves it exactly rather than widening it. |
| T-8-26 | Elevation of Privilege | `trustAllowlist` handling during the move | medium | mitigate | `warnUnknownTrustAllowlistEntries` moves verbatim; the allowlist gates trust escalation, so a relocation that changed its matching would silently grant or revoke escalation. Detector: `test/integration/plugin/` and the whole-system census. |
| T-8-16 | Elevation of Privilege | `NewManager`'s `ErrMissingVerbRegistry` guard (INV-EVENTBUS-11) | medium | mitigate | Kept at the tail of `NewManager`, behaving identically. Acceptance criterion runs the guard's test directly. Every in-repo `NewManager` constructor exercises it. |
| T-8-27 | Tampering | Harness bypassing `pluginsetup.NewPluginSubsystem` (INV-PLUGIN-18) | medium | mitigate | `TestWithInTreePluginsReusesSubsystem` reads `plugins.go` as text and asserts it contains no `plugins.NewManager(` call. D-07's frozen signatures make a harness rewire unnecessary; the acceptance criterion greps for it explicitly. |
| T-8-04 | Elevation of Privilege / Tampering | Lua vs binary parity across the finished split | **high** | mitigate | `event_emitter.go` remains untouched across the whole phase (asserted again here). `RegisterHost` and the four `Configure*` methods push dependencies into both host families identically, as they do today. Detector: `test/integration/pluginparity/` (7 specs). |
