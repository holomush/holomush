<objective>
Extract two small, cohesive CoreServer clusters into their own constructor-injected units:

- **`CommandHandler`** — `HandleCommand`, `executeCommand`, `executeViaDispatcher`,
  `emitCommandResponse` (deps: `dispatcher`, `cmdServices`, `presence`, `publisher`, plus the
  session-ownership preamble pair).
- **`LifecycleHandler`** — `Disconnect`, `recomputeSessionLiveness`, `runDisconnectHooks`
  (deps: `presence`, `disconnectHooks`, plus the preamble pair).

They ship together because both are small, both depend on `presence.Emitter`, and pairing them
makes the shared-collaborator resolution explicit in one reviewable change rather than implicit
across two.

Purpose: ARCH-01 / success criterion 1, continuing the cluster extraction 08-03 started.

Output: two new units with their separately-testable proof tests, and `server.go` reduced by seven
more methods.
</objective>

<planner_settled_decisions>
**This plan implements D-01's command-execution and session-lifecycle groups** — the second and
third of the four RPC groups D-01 names, continuing the extraction 08-03 began. Test scope follows
D-16: assert the new seams' structure and unit-testability, not behavior the integration suite
already pins.

**`presence.Emitter` shared by two units is not a conflict.** The matrix shows
`executeViaDispatcher` (command) and `Disconnect` (lifecycle) both touching `presence`. It is an
emitter — a collaborator, not shared mutable state owned by either unit. D-02 resolves it by passing
the same value into both constructors. Do not invent a coordinator, a mutex, or an ownership rule
for it; RESEARCH.md's conclusion for ARCH-01 is that there are no genuinely conflicted methods, and
this is the case that most looks like one.

**Same package, exported types** — same rationale as 08-03: exported so an external
`package grpc_test` can construct them (the SC1 proof), in-package so `command.Dispatcher`,
`command.Services`, and the unexported helpers need no export churn.
</planner_settled_decisions>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → `CoreService.HandleCommand` | Untrusted command text plus `player_session_token` cross here; `auth.ValidateSessionOwnership` (`server.go:405`) is the gate, and the dispatcher applies per-capability authorization downstream. |
| client → `CoreService.Disconnect` | Untrusted `session_id` + token; same ownership gate at `server.go:1550`. |
| server → event bus | `emitCommandResponse` publishes on behalf of a command; a duplicated or mis-attributed publish is observable by every subscriber. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-8-02 | Elevation of Privilege | `HandleCommand` and `Disconnect` session-ownership preambles | **high** | mitigate | Verbatim body moves — the preamble is an ordinary statement inside each body, so a byte-identical move cannot drop it. Both tasks require a line-for-line diff review. Detectors: `test/integration/auth/`, `test/integration/command/`, `test/integration/session/` under Task 3. D-15 turns a silent drop into a test failure. |
| T-8-17 | Spoofing | `emitCommandResponse` publish path | medium | mitigate | Behavior 3 pins a single publish with an explicit mock expectation count, so an accidental double-publish or a dropped publish fails the unit test rather than reaching the bus. |
| T-8-18 | Tampering | Disconnect-hook ordering | medium | mitigate | Hooks observe session teardown; reordering changes what each one sees. Controls: an explicit prohibition against reordering or parallelizing, plus Behavior 2 asserting the exact registration-order sequence rather than mere membership. |
| T-8-19 | Denial of Service | `NewCoreServer`'s nil-panic guard on `dispatcher` / `cmdServices` | low | mitigate | Left in place at the same construction point. A relocated guard that fires later would move a fail-fast wiring error into request-time. Task 1's action pins its position. |
| T-8-13 | Spoofing | `CoreServer`'s `corev1.CoreServiceServer` method set | low | mitigate | D-03 fixes the contract; Task 2 diffs the exported method-name list pre/post and asserts an empty proto diff. |
</threat_model>
