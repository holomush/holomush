<objective>
Extract the subscribe/stream-delivery cluster out of `CoreServer` into a `SubscribeHandler` that
takes only the 10 collaborators it actually uses, constructor-injected. `CoreServer.Subscribe`
becomes a thin delegator.

This is the largest and most collaborator-dense of the four ARCH-01 clusters (`Subscribe` alone
touches 8 of the 30 fields â€” the maximum fan-out in the whole matrix), which is why it gets its
own plan and goes first.

Purpose: ARCH-01 / success criterion 1. The bar is not "fewer lines in server.go" â€” it is that a
unit test can construct this handler with only its own collaborators. That property is the
deliverable.

Output: `internal/grpc/subscribe_handler.go` + its separately-testable proof test, with
`server.go` reduced by the moved cluster and the proto contract untouched.
</objective>

<planner_settled_decisions>
**Packaging: same package `grpc`, exported type.** CONTEXT.md leaves packaging to discretion; the
constraint is D-02's narrow-dependency rule, not the package boundary. Keeping the unit in
`package grpc` avoids exporting `SessionStreamRegistry`, `SessionStreamContributor`,
`SceneMuteChecker`, `WorldQuerier`, and the unexported helpers the cluster uses â€” churn that would
buy nothing. The **type** is exported so an external `package grpc_test` can construct it, which is
what makes "separately testable" provable rather than asserted. This mirrors
`RepoCharacterNameResolver` (`internal/grpc/character_name_resolver.go:25`), already an exported,
constructor-injected, `*CoreServer`-free unit living in this very package.

**This plan implements D-01's subscribe/stream-delivery group.** D-01 names four RPC groups to
extract behind the thin facade, continuing the `auth_handlers.go` precedent (16 methods already
split off and working). This plan takes the first group; 08-05 takes command and lifecycle, 08-07
takes current-state queries.

**Test scope follows D-16.** No up-front characterization or golden tests: the existing integration
and whole-system suites already ARE the characterization layer, which is why the ROADMAP success
criteria name them. The new tests in this plan assert the **new seam's structure and
unit-testability** â€” that the extracted unit is constructible and exercisable with only its own
narrow dependencies â€” rather than re-asserting behavior the integration suite already pins.

**Shared collaborators are not a conflict.** `focusCoordinator` and `streamContributor` are also
query-unit dependencies; `identityRegistry` is shared with the query unit; `worldQuerier` with auth.
Every one is a read-only interface or emitter, not shared mutable state â€” so D-02 resolves them by
passing the same value into both constructors. There is no coordination problem in the CoreServer
split; do not invent one.
</planner_settled_decisions>
