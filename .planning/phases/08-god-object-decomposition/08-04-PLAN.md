<objective>
Extract the plugin identity registry — `nameByID`, `activeByName`, `pluginRepo`, `retentionDays`
— out of `Manager` into an `IdentityStore` that owns its own `sync.RWMutex`. `Manager.NameByID`
and `Manager.IDByName` become one-line forwards.

This is D-06, and it goes first in Wave B because it is the coupling that makes a plain two-way
load/runtime split fail: these fields are *written* by the load family and *read* by the runtime
family, and today a single `m.mu` guards everything (`manager.go:95-99` says so explicitly).

Purpose: ARCH-02 / success criterion 2. Breaking the shared mutex is the prerequisite for the
load and runtime units (08-06, 08-08) to have genuinely disjoint state.

Output: `internal/plugin/identity_store.go` + its standalone test, with `Manager` three fields
lighter and one lock less overloaded.
</objective>

<planner_settled_decisions>
**Lock-split safety is settled, not open.** Research verified that `loadPlugin` takes and releases
`m.mu` in **four short, disjoint critical sections** and never holds it across an
identity-to-runtime boundary: the identity write is `manager.go:1171-1174` and the runtime commit
is `manager.go:1362-1369` — separate acquisitions with the lock released between them. Therefore
splitting the lock changes no lock ordering, introduces no nested acquisition, and preserves the
already-existing interleaving window (identity populated before `loaded`/`pluginHosts`) *exactly*.
This is the strongest de-risking finding in ARCH-02 and it is stated here so a reviewer does not
re-litigate it.

**Test scope follows D-16.** No up-front characterization or golden tests. The six behaviors below
assert the **new unit's** structure and standalone constructibility; the existing
`internal/plugin/identity_registry_test.go` and the whole-system census already pin the
Manager-level behavior and are this plan's oracle rather than something to duplicate.

**`retentionDaysSet` stays on `Manager`.** It is read only inside `NewManager` (`manager.go:232`)
to decide whether to apply the default of 3; it is option-plumbing, not identity state. The
*resolved* `retentionDays` value moves into the store.

**`m.hosts[TypeLua]` vs `m.luaHost` is NOT collapsed in this phase.** CONTEXT.md flags it as
behavior-adjacent; `luaHost` has fan-in 8 and participates in the `TestLoadPlugin` fallback. A
behavior-preserving phase is the wrong place to settle it. File a follow-up GitHub issue at phase
close; do not fold it in here.
</planner_settled_decisions>

|----------|-------------|
| plugin ULID → display name | `nameByID` resolves actor identity for audit rows and gRPC actor display. A wrong or missing resolution misattributes an action. |
| concurrent load / unload / read | `Manager.mu` today serializes every mutation; this plan gives identity its own lock. The boundary is between the identity lock's critical sections and `m.mu`'s. |
