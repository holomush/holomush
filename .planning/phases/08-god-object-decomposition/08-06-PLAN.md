<planner_settled_decisions>
**Field ownership boundary.** The runtime unit owns `loaded`, `inflight`, `pluginHosts`, `hostCaps`,
and the lock guarding them. `Manager` retains `hosts`, `luaHost`, `eventEmitter`, `policyInstaller`,
`registry`, `capVocab`, `aliasCache`, `aliasSeeder`, `verbRegistry`, `loadedOrder`, `pluginsDir`,
`trustAllowlist`, `gracefulDegradation`, `registerProvider`, `unregisterProvider` — those are
load-time wiring and go to the loader in 08-08.

**`RegisterHost` and `ConfigureEventEmitter` stay on `Manager`.** Both are registration/wiring
methods, not delivery. `RegisterHost` writes `hosts` (stays) *and* `hostCaps` (moves), so it remains
on `Manager` and forwards the capability-caching half into the runtime unit. Same for the four
`Configure*` methods: they push dependencies into hosts and stay where the hosts are.

**`Close` and `UnloadPlugin` are NOT in this plan.** They span all three units and are assigned to
the **load unit** in 08-08 — see that plan's settled decisions for the rationale. Do not
opportunistically move either here; that would fragment a decision that belongs in one place.

**D-20 is preserved structurally, not by discipline.** `event_emitter.go::Emit` is the single
chokepoint where both the Lua return-value emit path and the binary gRPC `EmitEvent` path hit the
manifest gates. This plan moves the *receiver* of `EmitPluginEvent` but does not edit
`event_emitter.go` at all — so there is no mechanism by which one runtime could end up on a
different gate. The acceptance criteria assert that file is untouched.
</planner_settled_decisions>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| plugin (Lua or binary) → host emit path | `event_emitter.go::Emit` enforces `actor_kinds_claimable`, `emits`, and `crypto.emits` manifest gates for BOTH runtimes. This plan moves the receiver of the `EmitPluginEvent` wrapper; the chokepoint itself is untouched. |
| AuthGuard → manifest crypto gates | `PluginRequestsDecryption` / `PluginCanReadBack` authorize decryption and read-back for a named plugin. Their receiver moves; their logic must not. |
| host → plugin dispatch | `DeliverEvent` / `DeliverCommand` / `BeginServiceDispatch` route to a named plugin's host. A wrong `pluginHosts` lookup delivers to the wrong plugin. |
