<objective>
The transaction & repository foundation the reviews identified as an unaddressed prerequisite (Codex Agreed-Concern A). Three source-grounded blockers make the drawn slice-1/slice-2 mechanism unimplementable as written, and this plan fixes them once, behavior-preservingly, before the version-guard repo rewrites (05-02/05-03) and the outbox (05-05+):

1. `Transactor.InTransaction` always calls `pool.Begin` (`transactor.go:28`) — it is NOT re-entrant, so a repo method that opens a tx while its caller already holds one nests/escapes; and the zero-row classifier's "locked follow-up read in the same tx" cannot reuse the caller's connection.
2. `ExitRepository.Create`/`Delete` (`exit_repo.go:60,178`) and `ObjectRepository.Move` (`object_repo.go:172`) open their OWN `pool.Begin` transactions, and `SceneRepository.AddParticipant`/`RemoveParticipant` (`scene_repo.go:40,57`) use `r.pool.Exec` directly — all four escape any ambient mutation transaction, so the outbox row would commit separately.
3. Delta-parity is unbuildable from command inputs because cascade IDs (the reverse-exit ID at `exit_repo.go:80`, cascaded children) are generated INSIDE the repository; service methods only see `error`. There is no `MutationDelta` contract for repos to return.

Purpose: The atomicity guarantee (MODEL-04) and the version guard (MODEL-03) both require every world write to enroll in one caller-owned transaction and to report the rows it actually touched. That is a repository-layer contract change, and it must land before the guard/outbox logic builds on it.
Output: re-entrant `InTransaction` + a re-entrant `withTx` seam; exit/object/scene writes refactored onto the ambient executor; the `MutationDelta` contract; redesigned repo write interfaces (`Delete(ctx, id, expectedVersion)`, writes returning `(*MutationDelta, error)`) with read-only reader views; regenerated mocks; all callers compiling. No behavior change.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| caller tx → repo write | A repo write that opens its own transaction escapes the caller's atomicity guarantee; the re-entrant seam closes this. |
| repo cascade → delta manifest | Cascade IDs generated inside the repo must be reported to the caller or the delta-parity manifest cannot match the committed rows. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-05-38 | Tampering | escaping repo transactions | high | mitigate | exit Create/Delete + object Move + scene writes route through the re-entrant seam / execerFromCtx; a `grep` gate + behavior tests forbid a surviving `r.pool.Begin`/`r.pool.Exec` write. A write that split the tx would leave an orphan outbox row (proven impossible in 05-05's atomicity test). |
| T-05-39 | Repudiation | non-reentrant tx double-Begin | high | mitigate | InTransaction reuses the ambient tx; a nested Begin would deadlock the locked follow-up read against a row the outer tx holds — the re-entrancy test proves the shared-connection path. |
| T-05-40 | Tampering | delta manifest cannot match rows | medium | mitigate | The `MutationDelta` contract makes repository-internal cascade IDs (reverse exit, children) reportable to the mutate() seam; the manifest is built from actual touched rows, not command inputs (bound by INV-WORLD-DELTA-PARITY in 05-12). |
</threat_model>
