<objective>
Close the round-3 correctness gap (Codex blocker #5 â€” the most substantive round-3 catch): build ONE atomic character-genesis application service and route ALL production character creation through it, so INV-WORLD-4 ("exactly one semantic envelope per successful externally-visible command") is actually TRUE for character creation.

**Why this plan exists (round-3 blocker #5):** the write fence protects `world.Service` (reader views) and raw SQL (AST fence), but THREE production paths create characters directly and would commit rows WITHOUT genesis envelopes: (1) registered creation â€” gRPC `CreateCharacter` â†’ `createCharacterAtomic` (`internal/grpc/auth_handlers.go:517-538`, with a non-transactional fallback at :521-526); (2) guest creation â€” `internal/auth/guest_service.go:32-35` declares its own `Create`-bearing repo interface and writes a character inside its own tx at :134-146; (3) bootstrap-admin creation â€” `internal/bootstrap/admin.go:85-98` calls `CharService.Create`, which reaches the concrete postgres repo through `CharRepoAdapter` (`internal/bootstrap/setup/adapters.go:25-43`). The AST SQL fence cannot catch these (the SQL stays in allowlisted `internal/world/postgres`); reader views cannot help (these paths hold a concrete or narrow writer). The fix is structural: one service owns character insert + binding + genesis envelope in a single transaction; the narrow interfaces LOSE their `Create` methods so no production package can insert a character any other way; and the 05-07 composition allowlist keeps concrete writer construction at composition roots.

**Round-4 B4 (atomicity scope correction):** round 4 found the round-3 "guest player+character+binding+envelope one commit" and "bootstrap player+character+role+envelope one commit" claims FALSE â€” `player_repo` (own r.pool) and `role_store` (own s.pool) do NOT enroll in world/postgres's private `txKey`, so only the character + optional binding + envelope are genuinely atomic (BindingRepository reads the world tx context). This plan NARROWS the atomicity claim to that sound unit, RESOLVES the FK-blocking hazard by ordering (guest commits the player before the genesis tx; bootstrap assigns the admin role after the genesis tx commits â€” so a role insert on a second connection never blocks on the outer transaction's uncommitted character FK), and DOCUMENTS orphan-player / admin-role-missing as accepted compensation gaps outside INV-WORLD-4. It deliberately does NOT introduce a shared exported transaction executor (round-4 B4 option a) â€” that would reopen 05-14's intentionally-unexported `txKey`.

This plan supersedes the round-2 "thread an OutboxWriter through CoreServer" design â€” the same end-to-end guarantee is delivered through the service constructed at `cmd/holomush/sub_grpc.go`, covering all three paths instead of one.

Purpose: A character that exists in the world but never appeared on the world feed is exactly the M2-class drift Phase 5 exists to close. This plan makes envelope-less character creation impossible by construction, not by convention.
Output: `auth.CharacterGenesisService` (insert + optional binding + genesis envelope, one re-entrant tx, fail-closed), all three creation paths rerouted, `Create` removed from the auth-side character-repo interfaces, composition-root wiring, behavior-preserving binding semantics.
</objective>
