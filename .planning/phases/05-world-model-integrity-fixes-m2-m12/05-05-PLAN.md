<objective>
Build the outbox foundation (MODEL-04, slice 2, interface-first): the outbox + `world_feed_counter` schema (migration 000050, now with the persistent feed-epoch columns), the semantic `Envelope` + `EnvelopeIntent` value types (in the cycle-neutral leaf package `internal/world/wmodel` — round-2 import-cycle fix, NOT `internal/world/outbox`), and — in `internal/world/postgres` (the writer boundary) — the same-transaction `WriteEnvelope` outbox row writer + the locked per-game `feed_position`+epoch allocator (with a bounded lock timeout). Placing the value types in `wmodel` and the write SQL in `postgres` resolves the Codex package-location contradiction (finding 6) AND the round-2 import cycle: `world`/`postgres`/`outbox` all depend on `wmodel` and never on each other for these types. This is the contract the mutation wrapper (05-06) and the relay (05-07) build against.

Purpose: The atomic feed requires (1) a place to durably write one envelope per command in the same tx as the state change, and (2) a gap-free, commit-ordered position allocated from a locked per-game counter — NOT BIGSERIAL, whose insert order does not equal commit order. Because the counter FOR UPDATE lock serializes all same-game writes, it is acquired as LATE as possible (envelope buffered, counter locked + row inserted right before commit) with a `lock_timeout`/`statement_timeout` bounding a stuck lock and surfacing the typed WORLD_FEED_LOCK_TIMEOUT (round-2 MEDIUM — a real task + timeout test, not prose). The persistent per-game epoch (round-2) lets a DB restore/backfill advance the epoch so stale positions are never replayed.
Output: migration 000050 (with epoch), the Envelope/EnvelopeIntent value types in wmodel, the tx-enrolled outbox store (WriteEnvelope) + locked counter+epoch allocator (bounded timeout) in `internal/world/postgres`, the always-run INV-WORLD-1 atomicity test, unit + integration tested.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| mutation tx → outbox row | The outbox write must be atomic with the state change; escaping the tx breaks the M2 fix. |
| counter allocation → concurrent writers | Two writers must never receive the same position (gap-free ordering). |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-05-12 | Tampering | outbox writer tx enrollment | high | mitigate | Writer uses `execerFromCtx`; a rollback-atomicity integration test proves a state change and its envelope commit or roll back together — a write escaping the tx would fail that test. |
| T-05-13 | Tampering | feed_position allocation | high | mitigate | Position comes only from the locked per-game counter (SELECT ... FOR UPDATE); the `(game_id, feed_position)` UNIQUE constraint makes any duplicate/insert-time allocation a hard DB error, not a silent gap. |
| T-05-14 | Spoofing | event_id dedup key | medium | mitigate | `event_id` is a fresh ULID from `core.NewULID()` and UNIQUE in the table; it is the `Nats-Msg-Id` dedup key. Hand-minted or reused ids are forbidden and would collide on the unique constraint. |
| T-05-15 | Information Disclosure | envelope payload | low | mitigate | Payloads are intent-level new-values-only (erasure-safe); no secrets or DEK material. ASVS V5 input validation applies to payload construction. |
</threat_model>
