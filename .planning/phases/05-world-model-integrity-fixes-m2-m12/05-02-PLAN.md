<objective>
Rewrite the location and exit repositories to version-predicated CAS with a locked follow-up read that classifies a zero-row result (conflict vs concurrent-delete vs not-found), and switch every write method from `r.pool.Exec` to `execerFromCtx` so it enrolls in an ambient transaction. This is the MODEL-03 mechanism for two of the four aggregates; characters and objects are the parallel plan 05-03.

Purpose: Today `Update` uses `r.pool.Exec` with no version predicate and collapses zero-row to `LOCATION_NOT_FOUND` — a stale-version write silently wins and reports success (M12). The guarded CAS plus the locked follow-up read is the exact NORMATIVE fix; the `execerFromCtx` switch is the single most under-appreciated change (it also sets up slice-2 outbox enrollment).
Output: guarded location_repo + exit_repo with a shared zero-row classifier, unit-tested for all three outcomes.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| concurrent replicas → same row | Two writers race the same location/exit row; the guard must reject the stale writer deterministically. |
| repo write → transaction | A write that escapes the ambient tx breaks the atomicity the outbox will depend on. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-05-04 | Tampering | version-predicated CAS | high | mitigate | Stale writer's UPDATE affects zero rows because its `version` no longer matches; the follow-up read classifies it as WORLD_CONCURRENT_EDIT — no silent overwrite. A forged/high `Version` cannot bypass CAS: it must equal the stored value, which only a genuine read produces. |
| T-05-05 | Repudiation | zero-row classifier | medium | mitigate | The locked follow-up read in the same tx removes the diagnosis race (conflict vs delete vs not-found is deterministic), so a lost update cannot be misreported as not-found. |
| T-05-06 | Tampering | execerFromCtx enrollment | high | mitigate | Every write routes through `execerFromCtx`; a write left on `r.pool.Exec` would escape the outbox tx silently — the acceptance criteria and a grep gate forbid it. |
</threat_model>
