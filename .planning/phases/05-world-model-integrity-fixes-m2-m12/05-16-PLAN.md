<objective>
Close the round-5 D-06 hole: the guest reaper and failed-guest cleanup delete characters through an FK cascade with NO tombstone, so a reaped guest's character enters the feed via genesis (05-15) at creation but its deletion is invisible — INV-WORLD-4 / delta-parity / feed-completeness are FALSE for a live production path. This plan is the deletion-side symmetric counterpart to 05-15's `CharacterGenesisService`: one atomic character-reaping service routes ALL guest character deletion through a tombstone-emitting world deletion path, so guest expiration cannot produce genesis-without-tombstone feed history.

**Why this plan exists (round-5 Codex, locked as D-06):** `guest_reaper.go` (`reap()` → `GuestCleaner.DeleteGuestPlayer`) and `guest_service.go` (`cleanupGuestPlayer` → `s.players.Delete`) both delete the guest PLAYER, relying on the `characters.player_id ON DELETE CASCADE` FK (`internal/store/migrations/000002_player_is_guest.up.sql:9`) to remove the guest's characters — with no character tombstone. The AST SQL fence (05-09) cannot catch this: the Go source contains `DELETE FROM players` (an auth-table write, not a world table), and the character deletion happens invisibly at the DB FK layer. Creation is fully covered (05-15 routes registered + guest + bootstrap-admin through the genesis service); deletion was not. This plan fixes it by construction, not convention.

**Design (mirrors 05-15, round-4 B4 discipline for the cross-pool boundary; hardened by round-6 R6-1/R6-2/R6-3):** a `CharacterReapingService` owns, in ONE re-entrant world transaction PER CHARACTER, the tombstone-emitting deletion of each of a guest player's characters — MIRRORING `world.Service.DeleteCharacter`'s cascade: `propertyDeleter.DeleteByParent("character", id)` (round-6 R6-3 — entity_properties has no FK to characters, so a bare delete orphans property rows), then guarded world `CharacterWriter.Delete` → tombstone `wmodel.MutationDelta`, then `OutboxWriter.WriteIntent` with the character delete/tombstone kind. It lists characters via the VERSION-SCANNING `ListByPlayer` (round-6 R6-1 — so the guarded CAS Delete has each character's stored version; the version-blind adapter list would make every delete a permanent conflict). It MARKS the player reaping before enumerating and the CharacterGenesisService REJECTS creation for a reaping player via a `SELECT reaping_at ... FOR UPDATE` (round-6 R6-2 — closing the snapshot-then-delete TOCTOU where a concurrently-created character was FK-cascade-deleted without a tombstone). The player-row DELETE stays on `player_repo`'s own pool and is ORDERED AFTER all character-tombstone txs commit, so the FK cascade has no un-tombstoned character left to delete. Orphan-player (player DELETE fails after character tombstones commit) is an accepted, documented compensation gap outside INV-WORLD-4 — symmetric to 05-15's orphan-guest-player gap.

**Why option (b) reaping-lock+genesis-reject, not option (a) single shared tx (round-6 R6-2):** a single PG transaction spanning enumerate + all character tombstones + the guarded player delete would close the TOCTOU, but it is PRECLUDED by the round-4 B4 two-pool boundary — `player_repo`'s player delete runs on its own pool and cannot enroll in `internal/world/postgres`'s private `txKey`. So the TOCTOU is closed at the CREATION side instead: mark reaping first, reject genesis for a reaping player, with the mark-UPDATE and the genesis `SELECT ... FOR UPDATE` serializing on the shared players ROW LOCK (cross-connection, same database).

Purpose: A character that appeared on the feed at creation (genesis) but vanished from the DB without a tombstone — even one created concurrently with reaping — is exactly the feed-completeness drift Phase 5 exists to close. This plan makes tombstone-less guest character deletion impossible by construction, under concurrency.
Output: the anti-TOCTOU substrate (`players.reaping_at` migration + a genesis reaping-reject guard + `MarkReaping`), `auth.CharacterReapingService` (per-character {property cascade + tombstone-emitting deletion} + ordered player delete, one re-entrant tx per character, version-bearing list, fail-closed), the guest reaper + failed-guest cleanup rerouted through it, composition-root wiring, a regression test proving no genesis-without-tombstone feed history, and a deterministic race test proving the TOCTOU is closed.
</objective>
