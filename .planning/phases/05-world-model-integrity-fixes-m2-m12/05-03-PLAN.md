Fill the CAS logic in the character and object repositories on top of the 05-14 foundation — the same guard, connection-reusing classifier, `MutationDelta` population, and struct-Version refresh as location/exit (05-02). This completes the MODEL-03 guard for all four aggregates. Depends on 05-14 (re-entrant tx, redesigned signatures, object-Move enrollment, regenerated mocks) and 05-01 (version column + WORLD_CONCURRENT_EDIT + MutationDelta type). This plan implements guard logic in already-redesigned bodies; it does NOT re-declare interfaces or regenerate mocks.

Purpose: Characters and objects are the other two aggregates the version guard must cover; `MoveObject`/`MoveCharacter` are the M2 movers slice 2 wires through the outbox, so their repo writes must be guarded and tx-enrollable first. The classifier's follow-up read MUST run on the caller's connection (05-14 re-entrant seam), never a fresh pool borrow (finding 14).
Output: guarded character_repo + object_repo with populated MutationDelta + refreshed Version, unit-tested for conflict / concurrent-delete / not-found + a pool-size-1 no-deadlock test.
</objective>

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |

| T-05-07 | Tampering | character/object CAS | high | mitigate | Stale move/update affects zero rows (version mismatch); the follow-up read surfaces WORLD_CONCURRENT_EDIT. A crafted `Version` cannot bypass CAS — it must equal the stored value a genuine read returned. |
| T-05-08 | Tampering | object Move enrollment | high | mitigate | Object Move routes through `execerFromCtx` so it commits atomically with the (slice-2) outbox row; a `r.pool.Exec` Move would split the transaction. Grep gate + acceptance criteria forbid it. |
| T-05-09 | Repudiation | zero-row classifier | medium | mitigate | Locked follow-up read in the same tx makes conflict/delete/not-found deterministic. |
</threat_model>
