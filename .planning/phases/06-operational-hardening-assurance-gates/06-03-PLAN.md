<objective>
OPS-03 (D-03/D-04): remediate the nats-server CVE and stand up a blocking
vuln-scan CI gate that would have caught it.

MECHANISM CORRECTION (flag explicitly — this is NOT a re-scope): D-04 named
govulncheck, but the researcher empirically confirmed the SPECIFIC advisory
GHSA-q59r-vq66-pxc2 is ABSENT from the Go vulnerability database (narrow claim
per review — other nats-server advisories DO exist in the Go DB; THIS one does
not), so `govulncheck ./...` passes on the vulnerable build. govulncheck alone
therefore CANNOT satisfy success-criterion #2 ("a build pinning a known-vulnerable
nats-server fails the gate"). The gate uses govulncheck (callgraph-aware
reachability, Go vuln DB) PLUS OSV-Scanner v2 (OSV DB, which ingests the GHSA nats
advisories) — OSV-Scanner is the leg that makes a pinned-v2.14.2 build fail and
provides the native `[[IgnoredVulns]]` allowlist.

EXCEPTION-POLICY SCOPING (review blocker — the allowlist covers only OSV-Scanner):
govulncheck has NO finding-suppression mechanism (upstream golang/go#59507), so an
accepted REACHABLE Go vuln cannot be allowlisted — it is a HARD STOP that must be
remediated (bumped/patched), never suppressed. The osv-scanner.toml `[[IgnoredVulns]]`
allowlist (with `id` + `reason` + `ignoreUntil` EXPIRY) applies ONLY to the OSV-Scanner
leg. This asymmetry is intentional fail-closed and is documented in the task and SUMMARY.

Purpose: close F8 (#4790) and give later phases a supply-chain gate.
Output: the nats bump, `task lint:vuln`, `osv-scanner.toml`, and a `vuln:` CI job.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CI vuln gate → merge decision | The gate is the control that stops a vulnerable dependency from merging; a fail-open gate is worse than no gate (false assurance) |
| tool install → CI runner | govulncheck/osv-scanner binaries execute in CI; an unpinned/tampered install is a supply-chain vector |
| allowlist (osv-scanner.toml) → gate output | An allowlist entry silently suppresses a finding; misuse hides a reachable CVE |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-06-3-01 | Spoofing/Elevation (fail-open) | task lint:vuln gate | high | mitigate | Two-tool gate; osv-scanner covers the Go-vuln-DB gap; pass/fail by EXIT CODE not stdout string (search-tools rule); pre-bump FAIL demonstrated before shipping |
| T-06-3-02 | Tampering | allowlist misuse suppressing a reachable CVE | high | mitigate | osv-scanner.toml seeded EMPTY; the nats CVE is NOT allowlisted; every future entry requires `reason` + `ignoreUntil` EXPIRY (reviewable in PR). The allowlist scopes to OSV-Scanner ONLY — govulncheck has no suppression, so a reachable Go vuln is a hard stop (fail-closed, not allowlistable) |
| T-06-3-SC | Tampering (supply chain) | osv-scanner / govulncheck install | high | mitigate | Blocking human legitimacy checkpoint (Task 1) pins version + SHA-256; CI installs checksum-verified, never `@latest` |
| T-06-3-03 | Denial | vuln-DB fetch flakiness makes the gate non-deterministic | medium | accept | Pin tool + versions; note in SUMMARY; a transient DB-fetch failure is re-runnable, not a silent pass |
