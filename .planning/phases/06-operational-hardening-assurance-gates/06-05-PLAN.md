<objective>
OPS-04 (D-05/D-06): fix the audit-DLQ replay CLI for external-NATS deployments
and replace its tautological coverage test with a genuine recovery assertion.

The F3 bug: the CLI reads `event_bus.game_id` from operator config; when that is
empty, `dlqConfigForGame("")` yields an empty-Subject config that `Defaults()`
fills to `internal.main.audit.dlq`. But the server persists its game_id (usually
a ULID) and publishes to `internal.<ULID>.audit.dlq`, so the CLI's prefix never
matches → every dead letter is counted Failed and nothing is recovered.

Fix (REDESIGNED per review): resolve the effective game_id MIRRORING the server's
own order at core.go:300-303 — `--game-id` override → `core.game_id` (the
`coreConfig.GameID` field, koanf tag `game_id`, loaded via
`config.Load(configFile, cmd, &cfg, "core")` exactly as core.go:125 does — NOT
`event_bus.game_id`, NOT the YAML root, and WITHOUT applying
`event_bus.Config.Defaults()`, which fills empty GameID with "main" and would
reintroduce the mismatch) → persisted DB value. The prior plan had this
inverted (override → DB → config), which diverges from the server whenever an
explicit `core.game_id` ≠ the DB value. Replace the tautological test (same `"main"`
on both seed and replay, embedded NATS) with a natstest-backed test IN cmd/holomush
that drives the REAL resolveGameID seam end-to-end and asserts both the failure
guard and recovery on a divergent-game path.

ROUND-2 REVIEW INCORPORATION (06-REVIEWS.md finding 13): every "ROOT game_id" /
"ROOT config" instruction is replaced with the exact source — `core.game_id`,
the `coreConfig.GameID` field loaded via `config.Load(configFile, cmd, &cfg, "core")`
(core.go:64,125) — so the executor loads the `core` config SECTION, not the YAML
root and not `event_bus`.

ROUND-3 REVIEW INCORPORATION (06-REVIEWS.md consensus): round-3 confirmed the executable
body is correct but flagged residual STALE steering — Task 1's title still said "configured
root game_id" (now `core.game_id`), and both verify commands masked test failures behind
`| tail` (now capture the exit code). The 06-PATTERNS.md DLQ section (which still prescribed
the REJECTED override → DB → cfg.GameID order) is corrected to override → core.game_id → DB.

ROUND-4 REVIEW INCORPORATION (06-REVIEWS.md Consensus Summary — LOW):

  coreConfig struct + Load; the `GameID` field is actually at `:70` and `config.Load(..., "core")` at
  `:125`. The citation is corrected (the action text was already right).
- **Config-section regression test (round-4 LOW):** the current tests exercise `resolveGameID` by
  passing values in, so `config.Load(..., "core")` itself is covered only by inspection. Task 1 now adds

Depends on 06-01 so the rewritten integration test runs against the final
