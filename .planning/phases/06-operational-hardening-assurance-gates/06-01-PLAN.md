<objective>
OPS-02 atomic core (D-01): convert `events_audit` from a single un-partitioned
table into a RANGE-partitioned table and — in the SAME plan/PR — change the shared
`writeAuditRow` INSERT so the audit write path still dedups correctly under the new
composite primary key.

REDESIGNED after cross-AI review (06-REVIEWS.md consensus blockers 1–3). Three
load-bearing corrections replace the prior DEFAULT-partition / timestamp-flip design:
- **No DEFAULT partition.** The prior design attached the old table as a permanent
  DEFAULT partition — non-viable: Postgres rejects the ATTACH when legacy rows fall in
  the current-month ranges, AND forbids `DETACH PARTITION CONCURRENTLY` while a DEFAULT
  exists (which would make 06-02's retention DDL invalid). The partitioned parent here
  has NO DEFAULT; legacy rows are re-homed by 06-02's one-time backfill.
- **A separate deterministic partition key, not the `timestamp` column.** The key is a
  NEW BIGINT column `event_ms = ulid.Time(id).UnixNano()`, derived purely from the
  immutable event ULID, so it is IDENTICAL on the live path and the DLQ-replay path (no
  cross-partition duplicate). The existing `timestamp` column is left UNTOUCHED (still
  JetStream store-time) so cold-history window filtering keeps its exact current
  semantics — no tier-boundary drift, no parity test needed.
- **Data-preserving down.** The down migration copies partitioned rows back into the
  restored un-partitioned table BEFORE dropping the partitioned parent, so a rollback
  loses no audit records.

Because `writeAuditRow` is shared by the live projection (`projection.go`) AND DLQ
replay (`replay.go`), the migration PK change, the conflict-target change, and the new
event_ms column are ONE indivisible unit. This plan also trips the mandatory
crypto-reviewer surface (projection.go + events_audit migration), gated by a blocking
pre-push checkpoint (Task 3).

Output: migration `000052_*` (partition-swap, idempotent, data-preserving down) plus
the `writeAuditRow` idempotency fix, proven by an integration test that the same event
through live + replay produces exactly one row under the composite PK.
</objective>
