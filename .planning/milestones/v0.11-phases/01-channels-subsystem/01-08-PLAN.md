<objective>
Wire the plugin half of live delivery onto the substrate landed in 01-02: implement `QuerySessionStreams` (session-establishment subscription) and use the `stream.subscription` capability for mid-session `join`/`leave` deltas (LIVE_ONLY). This is the CHAN-01/CHAN-02 live-delivery path.

**Guest auto-join mechanism (RESEARCH landmine #4 resolution — chosen shape):** guest auto-join is served by **unioning the seeded default channels into `QuerySessionStreams`, NOT by writing a membership row at session establishment.** `QuerySessionStreams` returns the deduplicated union of (a) the character's non-banned memberships and (b) `store.ListDefaultChannels` (the `is_default=true` set seeded in 01-03), minus any default the character is banned from. A guest — a session with zero membership rows — therefore receives exactly the seeded default channels. This keeps membership resource-side (no principal RPC, no session-establishment write) and plaintext (D-04). Seeding (01-03) is upstream of this serving via the transitive DAG (01-08 → 01-05 → 01-03).

Purpose: a connected member sees channel messages in real time; a guest is auto-joined to the seeded channels at connect; joining mid-session starts live delivery without reconnect; leaving stops it.
Output: `SessionStreamsHandler` impl (memberships ∪ ListDefaultChannels) + join/leave stream mutation + manifest `session_streams`/`requires`; unit tests green (the full member-vs-non-member delivery assertion runs in the 01-09 integration).

**SCOPING-RISK dependency (Landmine 2):** This plan consumes 01-02's substrate, which the cross-AI review sharpened into concrete work: 01-02 now (HIGH-1) makes the manager ACCEPT the RELATIVE `channel.<id>` session streams QuerySessionStreams here returns (no longer dropped), (R3-A) fences EACH plugin contribution at SESSION ESTABLISHMENT with the SAME namespace guard as mid-session (`pluginauthz.AuthorizePluginStreamContribution` inside `Manager.QuerySessionStreams`) — so core-channels' own `channel.<id>` is delivered but a foreign/system/`events.`-prefixed ref would be dropped, (HIGH-2) implements real `LIVE_ONLY` in `AddStreamWithMode` (so the mid-session `AddStream(..., LIVE_ONLY)` calls here actually work and don't flood history), and (HIGH-3 / R2-B) gates `AddSessionStream`/`RemoveSessionStream` with the SAME concrete-stream authz fence, scoped to core-channels' OWNED `channel` emits domain. **R2-A subject-form contract (verified against source):** 01-02's `AuthorizeStreamSubscribe` mirrors the read guard and REJECTS a pre-qualified `events.` subject with `STREAM_NOT_RELATIVE`, then `eventbus.Qualify`s the relative ref. So the AddStream/RemoveStream calls here MUST pass the domain-RELATIVE `channel.<id>` (via `relativeChannelStream(id)`), NOT the full `dotStyleChannelSubject` — if this plan passed the full subject the guard would reject core-channels' OWN join. Both downstream registry paths qualify idempotently (`applyFilterCtrl`→`toSubject`→`Qualify` for mid-session; `computeInitialFilters`→`toSubject` for establishment), so the qualified filter still resolves to `events.<game>.channel.<id>` and matches the emit subject. If 01-02's mid-session serving (`stream.subscription`) still landed only partially (per its documented degradation fallback), Task 2 here degrades to "join takes effect on next session-stream refresh" and MUST be flagged in the SUMMARY — do NOT silently drop it. Task 1 (session-establishment via `QuerySessionStreams`, delivered through the HIGH-1-fixed manager path) is the delivery floor and depends only on 01-02 Task 1.
