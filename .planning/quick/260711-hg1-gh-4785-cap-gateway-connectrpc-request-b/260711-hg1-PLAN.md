# Quick Task 260711-hg1 — Cap gateway ConnectRPC request-body size

## Problem

The public web gateway's ConnectRPC handler (`internal/web/server.go`) builds its
handler with no `connect.WithReadMaxBytes`. connect-go v1.20.0 only applies its
body `LimitReader` when `readMaxBytes > 0` (`protocol_connect.go:1119`), so an
unset value reads the **entire request body into memory**. A single
unauthenticated POST with a large body can OOM the gateway. The core gRPC server
already caps inbound at 4 MiB (`internal/grpc.MaxRecvMsgSize`); the public
surface — the higher-risk one — is uncapped. `ReadHeaderTimeout` (10s) is set but
`ReadTimeout` is not, leaving slow/large bodies unbounded in time as well.

## Acceptance (from issue #4785)

- Add `connect.WithReadMaxBytes(4<<20)` (match the gRPC cap).
- Add an `http.Server.ReadTimeout`.
- Regression test: an oversized body returns `resource_exhausted` rather than
  buffering.

## Approach

Single-file change in `internal/web/server.go` + one regression test in
`internal/web/server_test.go`.

1. Add package const `maxRequestBytes = 4 * 1024 * 1024` (4 MiB), documented as
   matching `internal/grpc.MaxRecvMsgSize` and citing GH-4785.
2. Pass `connect.WithReadMaxBytes(maxRequestBytes)` as the first handler option
   to `webv1connect.NewWebServiceHandler`.
3. Add `ReadTimeout: 30 * time.Second` to the `http.Server` literal.

### Streaming-safety note (verified, not assumed)

`ReadTimeout` is safe for the long-lived server-streaming `StreamEvents` RPC.
Under HTTP/2, `x/net/http2` (server.go:1987-1990) disarms the connection-wide
read deadline after headers and arms a **per-stream** timer whose handler
(`onReadTimeout`, server.go:1846-1852) closes only the **request body reader**
(`st.body.CloseWithError`), never the response writer. Server-streaming reads its
single request message before the handler starts writing, so the body is at EOF
when the timer fires — the response stream is unaffected.
