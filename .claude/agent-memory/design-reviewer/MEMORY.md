# design-reviewer agent memory

This file accumulates HoloMUSH-specific patterns of good and bad designs
discovered during adversarial design review. Entries are added by the agent
itself after completing a review.

Keep under 200 lines. Curate — don't hoard.

## Common spec weaknesses in this codebase

- [Round-2 fixes can introduce orthogonal contradictions](feedback_round2_fix_introduces_contradiction.md) — sweeping R2 generalizations flatten over earlier considered decisions in other sections; grep enumeration items
- [Amendment rolls back a just-landed sibling spec](feedback_amendment_contradicts_just_landed_sibling.md) — multi-epic chain Phase-N+1 amends master-spec contract that Phase-N sibling just-shipped (with code-level docs + meta-tests). Cross-check against (a) sibling-spec prose, (b) code doc comments, (c) `TestSpec*Amendments*` meta-tests, (d) parent decomposition spec's MUST decisions, (e) threat-model defense-in-depth rationale before accepting
- [Universal stamp-contract vs per-kind invariant](feedback_universal_stamp_contract_vs_per_kind_invariant.md) — universal "every stamp site MUST X" + per-kind invariant carving out kind D = kind-D production stamps fail at cutover; grep all kind-D sites
- [oops vs gRPC code conflation](feedback_oops_vs_grpc_code_conflation.md) — HoloMUSH specs often call oops codes "gRPC codes"; flag wording, do not block if pattern matches existing codebase convention
- [grpc FromError message rewrite on wrapped statuses](feedback_grpc_fromerror_message_rewrite.md) — `st.Message()` on a status reached via `errors.As` is the OUTER err.Error(); flag "verbatim" / "wire-equivalent" claims that ignore this
- **"Mirrors existing technique X" claims need verbatim verification.** When a spec says "use the same technique as `Y`", read both sides and diff them. Authors paraphrase or reconstruct from memory and produce structurally different (broken) variants. Seen 2026-04-25 in session-workspace-isolation: spec claimed to mirror `Taskfile.yaml:521-530` gowork's `.jj/repo` resolution, but the provided snippet computed a different path that failed when invoked from a worktree.
- **Shell-snippet error handling in spec docs is rarely verified.** Patterns like `var=$(cmd | tail -n 1) || return` (bash) and `set var (cmd | tail -n 1); or return $status` (fish) do NOT detect failure of `cmd` — the pipeline status is the LAST command's status. Run the snippet before believing it. Seen 2026-04-25.
- **Spec "verifiable by …" criteria with overly-broad jj revsets.** `jj log -r 'all()'` returns full history; for experiment-level verification prefer `mutable()` or scoped revsets. Not blocking but degrades the diagnostic value.
- **Stated rationale for shell idioms can be wrong even when the idiom works.** A spec may correctly include a `cd` or `cd && cmd` block but provide an incorrect *reason* (e.g., "needed because X requires Y" where Y is false). This is documentation rot risk, not a correctness blocker. Verify rationale claims by running the inner command without the `cd` to see if it really fails. Seen 2026-04-25 in session-workspace-isolation r3: spec claimed `cd "$MAIN_REPO"` was needed for `jj git fetch`, but `jj git fetch` works from any workspace because it resolves repo storage via `.jj/repo`.
- [Check existing RPCs before "new RPC" proposals](feedback_check_existing_rpcs.md) — auth/session/check specs often duplicate existing RPCs (e.g., `WhoAmI` vs `WebCheckSession`); always grep proto + handlers
- [Extending an existing RPC: audit callers for contract-shape dependencies](feedback_extending_rpc_check_callers.md) — when a spec extends an RPC's failure-path contract (error → `authenticated=false`), grep all current callers; web client try/catch patterns often depend on the throw
- **"ctx carries an actor at boundary X" claims need a full call-graph trace.** Specs that propose authenticating an actor pulled from `core.ActorFromContext(ctx)` at a specific boundary MUST be checked against EVERY production caller of that boundary, not the obvious one. `core.WithActor` is often called *after* the boundary call returns (for use in a downstream `EmitPluginEvent`), not before. Seen 2026-04-25 in plugin-actor-claim-auth: spec's §3.3.4 issuance flow assumed `Host.DeliverEvent`/`DeliverCommand` ctx carries an actor, but `subscriber.go:104-150` and `dispatcher.go:118-310` both stamp the actor only after the Deliver call. The spec's "verbatim cascade preserved" branch was dead code at every production call site.
- **Migration scope: enumerate ALL in-tree plugins that hit the new gate, not the marquee one.** Specs adding a manifest gate often name one plugin (the binary one with provenance) and miss Lua plugins that emit through a different code path. Always check BOTH `events:` declarations in `plugin.yaml` (subscriber-driven emits) AND command capability `action: emit` (dispatcher-driven emits) across every `plugins/*/plugin.yaml`, then read the corresponding `main.lua` / `main.go` to confirm whether character-driven emits actually happen. Seen 2026-04-25 in plugin-actor-claim-auth: r1 named only `core-scenes` and missed `core-communication`; r2 named both but missed `echo-bot` (Lua plugin subscribed to `events: [say]` that emits a follow-up `say` for non-plugin actors).
- **In-tree manifest filename is `plugin.yaml`, NOT `manifest.yaml`.** The `Manifest` Go type's docstring confirms (`internal/plugin/manifest.go:69, 291`). Specs that say "update `<plugin>/manifest.yaml`" are referring to a path that doesn't exist; the acceptance criterion is unverifiable as written. Always grep `internal/plugin/manifest.go` for the canonical filename before approving any acceptance criterion that names the manifest path.
- **Lint heuristics described in spec acceptance criteria need a verbatim 3-plugin walkthrough.** When a spec proposes "lint check that flags any plugin with property X", apply the proposed predicate to each in-tree plugin yaml in your head and check the result. Seen 2026-04-25 in plugin-actor-claim-auth r3: criterion 8's heuristic checked `action: emit` capabilities OR non-empty `events:`; this caught `core-communication` (capabilities) and `echo-bot` (events:) but silently passed `core-scenes` (top-level `emits: [scene]`, but command capabilities are `action: write` and no `events:`). The spec's listed migration scope had core-scenes in it but the proposed lint check would not enforce its continued correctness. The fix surface is "evaluate the proposed lint heuristic against every plugin in the migration scope" — if any plugin in the scope is not flagged by the heuristic when it should be, the heuristic is wrong, not the scope.
- **Migration-section consistency: every section that lists migrating plugins must list ALL of them.** Specs frequently fix a "missed plugin" finding in the design section but forget the same fix in §migration-path / §release-notes / §risks. Round 2 of plugin-actor-claim-auth surfaced echo-bot as missed; round 3 fixed §3.2, §6, criterion 8 but left §9 step 1 still naming only `core-scenes`. Always cross-grep the spec for `core-scenes` (or whichever was the marquee name) AFTER any migration-scope edit, and ensure each occurrence either references the full list or correctly scopes itself.
- **Lint heuristic enumeration claim ≠ heuristic logic.** When a spec writes "this heuristic catches X via path A and Y via path B and Z via path C", the FAILURE mode is that the conjunction in the heuristic doesn't actually evaluate to true on Z's manifest. Re-evaluate the heuristic predicate term-by-term against EACH plugin in the enumeration's actual on-disk yaml — do not trust the prose. Round 4 of plugin-actor-claim-auth: criterion 8's heuristic required `emits:` non-empty AND `(commands: OR events: non-empty)` AND missing-character-claim; the spec's own enumeration claimed it caught echo-bot "via events + emits in handler" but echo-bot's `plugin.yaml` has no top-level `emits:` field at all (the emits happen at runtime in main.lua), so the conjunction short-circuits to false. The heuristic operates on manifest static state, not runtime Lua handler behavior. When evaluating "does heuristic H catch plugin P", read P's plugin.yaml and check each conjunction term in the actual on-disk manifest — never accept the spec author's enumeration without verifying each entry.
- [Taskfile shell idioms must be verified against mvdan/sh](feedback_taskfile_shell_idioms.md) — go-task's inline `cmd:` runs under `mvdan.cc/sh/v3/interp`, not `/bin/sh`. `exec N>FILE` for N>=3 is rejected by mvdan/sh; fd inheritance into subprocess `task other-task` calls is fds 0/1/2 only (no `ExtraFiles`). Verify empirically (one-Taskfile fixture) before approving any spec with shell tricks in Taskfile cmds. Seen 2026-04-26 in pr-prep-concurrency-safety: spec proposed `exec 9>"$LOCK"; flock -n 9; task pr-prep:run` — empirically fails on first line.
- [go-task `internal: true` semantics](feedback_go_task_internal_semantics.md) — internal tasks are CLI-blocked by go-task itself (any `task <name>` invocation, even from a subshell, exits 202 with "Task is internal") AND hidden from both `--list` and `--list-all`. Layering a shell-level "bypass guard" on top is dead code: go-task's gate fires first. Seen 2026-04-26 in pr-prep-concurrency-safety r2: harness calls `exec task pr-prep:run` from `sh -c` block, but `pr-prep:run` is `internal: true` — lock acquires, info written, body never runs. Plus invariant I-10 asserted "MUST appear in --list-all" which is structurally false for internal tasks.
- [go-task wraps user exit codes by default](feedback_go_task_exit_code_wrapping.md) — go-task collapses any user-cmd non-zero exit to 201 ("Command execution error") unless `--exit-code`/`-x` is passed at every nested invocation layer. Specs that assert literal exit-code passthrough invariants ("harness exits 42 when stub exits 42") will fail. Seen 2026-04-26 in pr-prep-concurrency-safety r3: I-6 propagates_exit_code unsatisfiable as drafted, plus two test-row assertions of "exit 1" that empirically observe exit 201.
- [go-task `exec` inside one cmd does NOT short-circuit subsequent cmds](feedback_go_task_exec_does_not_short_circuit_cmds.md) — `[- cmd: detect && exec other, - cmd: fallback]` runs BOTH on success path; each `- cmd:` is its own subshell. Bare `exit 0` same. Only non-zero exit stops cmd2 (and propagates as wrapped 201). Seen 2026-05-14 in pr-prep-docs-fast-lane r2 §4.3.1.
- **"Existing field" claims about proto must be verified against the actual `.proto` file, not against comments or adjacent code.** Specs have called proto fields "existing" because a column with the same name exists on a database table or a header with that name exists. They are not the same wire-format surface. Always grep the proto. Seen 2026-04-25 in event-payload-crypto-design: spec marked `codec` as "EXISTING" on the `Event` proto; only the audit table column and NATS header have that name, the proto Event has no codec field.
- **Specs that build on a substrate spec must explicitly state which substrate abstractions they supersede.** When a new design introduces types that overlap or replace abstractions defined and shipped in a prior spec (e.g., `KeyProvider`/`KeySelector` from one spec, replaced by `Provider`/`DEKManager` in the next), the new spec must call out the supersession. Otherwise the implementation has two competing models in the same package. Seen 2026-04-25 in event-payload-crypto-design vs jetstream-event-log-design.
- **"Before X returns" / "synchronously" / "atomically" claims for cross-replica operations need a specified protocol.** NATS core publish, JetStream publish-with-ack, and request-reply have different consistency guarantees. A spec that says "MUST invalidate caches in all replicas before the operation returns" without specifying the ack mechanism is asserting an unprovable invariant.
- **Global negatives ("MUST NOT be written to disk anywhere ever") tagged as "Unit + Lint" are usually too vague to enforce.** Unit tests cannot prove a global negative; lint rules need concrete forbidden-symbol/forbidden-call lists. Force the spec to enumerate the actual checks.
- **Invariants whose AAD or signature includes proto-marshaled bytes must specify the canonicalization rule.** Proto serialization is not deterministic by default; if both encrypt-side and decrypt-side don't use the same canonical encoder a subtle library upgrade silently breaks decryption. Spec must name the canonicalization function.
- **Schema additions referenced in spec body but not in the data-model section.** When a spec mentions new columns on existing tables in flow diagrams or invariants but its data-model section only adds a NEW table, the migration for the additions is undefined. Plan-writers will invent column types. Seen 2026-04-25 in event-payload-crypto-design: `events_audit.dek_ref/dek_version` referenced but never declared in §4.
- **Lifecycle invariants asserted as MUST that the AuthGuard cannot satisfy.** When a spec promises "previous-tenure player retains decryption" but the access decision tree only matches the current binding, the invariant is unprovable. Walk every assertion against the access-control branches before believing it.
- **Fix-text in revisions can introduce NEW factual errors about repo symbols/files.** When a spec author addresses a "list the affected call sites" finding in round N, the resolution paragraph may name additional symbols (e.g., "and also delete `IdentityKeyProvider`") or wrong file locations (e.g., "from `codec.go`" when the symbol is actually in `publisher.go`). Always re-verify the fix-text against `rg` even when the prior round's finding was about the same area. Seen 2026-04-30 in event-payload-crypto-phase2-substrate r2: round 1 asked the author to list `KeyProvider`/`KeySelector`/`KeyLabel` call sites; round 2 added a "Phase 3 scope addition" paragraph that named `IdentityKeyProvider` (does not exist in repo) and located `identityKeySelector` in `codec.go` (it lives in `publisher.go:392`).
- **"Compile-time enforced" claims about proto serialization need to match the actual ruleguard scope.** A spec may claim "bare `proto.Marshal` is compile-time prevented by ruleguard rule X" when X actually only fires for arguments of a specific type unrelated to the proto message in question. Trace the ruleguard predicate to confirm what the rule actually catches. Seen 2026-04-30 in event-payload-crypto-phase2-substrate r2: claim that `dek_no_serialize` ruleguard prevents bare `proto.Marshal(event.Actor)` — but that rule's `Type.Is(...)` filter is on `*dek.Material`/`dek.Material`, not `*Actor`, so it would not fire. The defense is the runtime byte-equality test, not compile-time.
- [golangci-lint module-plugin package name claim](feedback_module_plugin_package_name.md) — module plugins do NOT require `package main`; canonical upstream example uses `package linters`. Verify against `golangci/plugin-module-register/register.go` and `golangci/example-plugin-module-linter`.
- **Lint-rule scope migrations need to account for the OLD rule's de-facto exclusions.** A rule that runs through `gocritic` inherits gocritic's `_test.go` exclusion in `.golangci.yaml`. Migrating that rule to a top-level linter in `linters.enable` LOSES the exclusion unless explicitly re-added. Always grep for callsites in `_test.go` to estimate the regression surface. Seen 2026-05-01 in go-analysis-migration: spec's `ulidmakeforbidden` declared "Allowlist: none (repo-wide)" but the codebase has 1300+ `ulid.Make()` callsites in `_test.go` that the existing rule (via gocritic) silently lets pass.
- [golangci-lint v1 vs v2 exclusion schema](feedback_golangci_v1_vs_v2_schema.md) — `.golangci.yaml` v2 uses `linters.exclusions.rules`, NOT v1's `issues.exclude-rules`. Always check the file's `version:` line before approving any exclusion-block YAML in a spec. Seen 2026-05-01 round 2 of go-analysis-migration: fix-snippet for the `_test.go` ulidmakeforbidden exclusion used v1 syntax against a v2 config, defeating the very fix it was meant to add.
- **"Cost" / callsite-count paragraphs need verbatim `rg` audit, including test-file impls and constructor seams.** Specs that change a substrate interface signature commonly miss (a) test stubs that implement the interface (`*_test.go` Codec/Provider impls also need the new signature), and (b) constructor seams that produce instances of the modified type (e.g., adding `Version uint32` to `codec.Key` requires updating `dek.Material.AsCodecKey` since it produces `codec.Key{}` literals, NOT just the field declaration). Seen 2026-05-02 in event-payload-crypto-phase3a-grounding: doc said "five callsites in codec.go + publisher.go + subscriber.go" but reality was 3 impls + 4 prod callsites + 4 test callsites + an `AsCodecKey` constructor needing signature change. Run `rg "func \(.*\) (Encode|Decode)\("` AND `rg "\.(Encode|Decode)\(ctx"` AND find any `Method-that-returns-the-modified-type` constructor before accepting a callsite count.
- **Substrate signature changes that touch both directions (Encode AND Decode) need decode-side wiring stated.** When a spec extends `Codec.Encode/Decode` with a new parameter (e.g., `aad []byte`), only the encode-side caller is usually described because that's where the new data is built. Decode callers also have to pass *something*. If the substrate has live Decode callers (subscriber, history hot tier, plugin audit consumer), the spec must state what value they pass in the interim — typically `nil` if `IdentityCodec` ignores. Seen 2026-05-02 in event-payload-crypto-phase3a-grounding: Decision 1 specified encode-side `aad.Build` but did not name what the 3 production Decode callsites pass during Phase 3a (when sensitive decode lands in 3b).
- **"Pseudocode call shape" in master spec doesn't validate against actual interface.** When master spec §X shows a function call with rich arguments (e.g., `abac_engine.Evaluate(subject, action, resource, attributes={...})`), grep the actual interface in code BEFORE accepting any spec that imports the interface as-is. Real `types.AccessRequest` had only `(Subject, Action, Resource string)` — no `attributes` field — so master spec §7.2 Branch 3's call shape was fictional. Grounding doc imported the real `accesstypes.AccessRequest` without acknowledging the gap. Fix surface: every spec that declares `<interface>.<method>(<args>)` should verify that `<args>` matches the real signature in the named package, not the master spec's prose. Seen 2026-05-02 in event-payload-crypto-phase3b-grounding finding 1.
- **Naming collisions between new package types and existing package types in the same import set.** When a new package introduces a type whose name already exists in a package the new package's consumers will import (e.g., `authguard.Subject` struct vs `eventbus.Subject` string), every callsite that imports both will have two `Subject`s in scope. Grounding docs that propose new packages should grep the consumers' import set for namespace conflicts BEFORE the type names are baked in. Seen 2026-05-02 in event-payload-crypto-phase3b-grounding finding 2: spec proposed `authguard.Subject` while `eventbus.Subject` is the JetStream filter type used in the very same `OpenSession` signature.
- **Constructor-list cross-section consistency.** A grounding doc that enumerates `New<Kind>Subject(...)` constructors in Decision A must enumerate the SAME set in every later decision that references them. Seen 2026-05-02 in event-payload-crypto-phase3b-grounding finding 3: Decision 1 declared four constructors and a 4-non-Unknown enum; Decision 2 named five constructors including `NewSystemSubject` with no enum value, no §X branch, no boundary-table row.
- **"Embedded `<other-package>.Decision` for trace data" rationale needs to match the actual struct shape.** Specs often write rationale paragraphs like "the foreign engine's Decision is embedded only when consulted, for trace data" but the struct definition has only flat fields. Trace data flow is undefined. Either the struct needs `ABACDecision <type>` (or pointer or embedded), or the rationale paragraph drops. Seen 2026-05-02 in event-payload-crypto-phase3b-grounding finding 4.
- **Cost-line for test-file edits frequently misdescribes the actually-load-bearing assertion.** When a wire-shape change ripples into an integration test, the cost-line author often says "byte-equality assertion shifts from X to Y" without checking that the existing assertion's invariant is preserved by the proposed shift. Always trace the assertion line-by-line under the new shape. Seen 2026-05-02 in event-payload-crypto-phase3b-grounding finding 5: spec said INV-21 assertion shifts from `msg.Data` to `envelope.Payload`, but `audit.projection.go:281` writes `msg.Data()` to the audit row — both stay byte-equal under the new shape; shifting to `envelope.Payload` REGRESSES INV-21.
- **TOCTOU defense paragraphs need to specify in-process plaintext-residency contract.** When a spec describes a permit→decrypt→enqueue-audit→stamp-metadata_only flow with a "stamp metadata_only=true on enqueue failure" defense, it does NOT specify what happens to the decrypted plaintext in process memory at the moment of the stamp. INV-9 ("subject not in DEK MUST NOT receive plaintext") is wire-testable but the in-process residency window is undocumented. Spec must state operation order: (a) AuthGuard first / decrypt only on permit, OR (b) decrypt first / overwrite-on-deny, with a unit-test invariant naming the chosen order. Seen 2026-05-02 in event-payload-crypto-phase3b-grounding finding 8.
- **Sibling type signature drift in same package.** When a spec proposes two new sibling types in the same package (`dek.Cache`, `dek.ParticipantsCache`) with conceptually identical methods (`InvalidateContext`), readers in different sections of the spec independently propose different signatures for the same method (one takes a struct, the other takes split string args). Always grep the spec for the method name and reconcile EVERY occurrence to one signature, including any pseudocode/protocol-flow callsites. Seen 2026-05-02 in event-payload-crypto-phase3c-grounding finding 1.
- **Single-replica self-pill failure mode.** Specs that propose strict-N + probe + pill protocols on a "self-counts-as-member" model usually say "single-replica deployments degenerate to N=1; the contract is identical." The CONTRACT is identical, but the FAILURE MODE is materially worse: in N≥2 a hung member gets pilled and cluster proceeds with N-1; in N=1 the member self-pills and the cluster self-immolates. Always require an explicit "Coordinator MUST NOT issue a pill targeted at its own MemberID" invariant + test, separate from the general probe-and-pill flow. Seen 2026-05-02 in event-payload-crypto-phase3c-grounding finding 4.
- **Performance-only substrate must say so explicitly.** When Phase X proposes new substrate above Phase X-1 and the only correctness invariant satisfied is one that Phase X-1 already satisfied differently, the new substrate is performance work. Spec must either (a) state "Phase X is performance optimization; correctness is unchanged from Phase X-1" and re-justify the substrate cost on that basis, OR (b) declare the new correctness-binding invariant the substrate enables. Without this, ~14 implementation tasks chase a ghost. Seen 2026-05-02 in event-payload-crypto-phase3c-grounding finding 5: Phase 3c "Full scope" caches participants but every new INV-53..58 covers protocol mechanics, none binds the load-bearing "after Add() returns, every replica's Participants(...) reflects the new participant within timeout" promise the substrate exists to deliver.
- **Authority-boundary ambiguity for primitives shared across consumers.** When a spec puts a primitive (`ProbeAndPill`) on a shared package (`cluster.Registry`) with the rationale "future consumers can reach for the same primitive," but then defines a per-call rate-limit on the *consumer* (`Coordinator`) rather than the primitive, the rate-limit is bypassable: any other consumer (operator @evict-member, future split-brain detector — both named in the spec) calls Registry directly and floods the wire. Pin rate-limits on the primitive, OR document explicitly "rate-limit is per-caller; future callers MUST also rate-limit." Seen 2026-05-02 in event-payload-crypto-phase3c-grounding finding 2.
- **Payload schema field reuse across actions with mismatched semantics.** When a spec defines a single payload schema for a multi-action protocol (`{old_version, new_version, action}`), action-specific semantics may not map to all fields. `Add()` on a vN active-version doesn't have an "old_version" — it has an "active_version that was just mutated." Plan-writers populate `old_version=0` (zero-value) → receive-side cache lookup keys on `Version: 0` → silent no-op eviction → the load-bearing invariant fails. Either rename the field to be action-neutral (`active_version`) or define per-action sub-schemas. Seen 2026-05-02 in event-payload-crypto-phase3c-grounding finding 3.
- **"Documentary" invariants are usually lintable.** When a spec assigns test-class "Documentary" to an invariant with rationale "absence-of-thing property," push back: most negative-space properties are testable via `gorules` ruleguard, semgrep, or `go/analysis` AST walker (e.g., "no subtraction of remote-sourced time field from local clock"). Force the spec to enumerate the actual lint check; if no enforceable check exists, the invariant is too vague. Seen 2026-05-02 in event-payload-crypto-phase3c-grounding finding 1 (non-blocking): INV-58 "no cross-host wall-clock comparison" is lintable.

- **Decomposition specs that lock substrate decisions can quietly contradict the master spec they cite.** When a meta-spec says "master spec §X remains authoritative" but then introduces a substrate decision that depends on a property the master spec explicitly denied, the contradiction surfaces in the dependent sub-epic spec (often weeks later) rather than at decomposition review. Always grep the master spec for the *negation* of any defense-factor / capability / mechanism the decomposition spec relies on. Seen 2026-05-07 in event-payload-crypto-phase5-decomposition Finding 1: Decision 6 listed "Shell access to host" as a defense factor; master spec §5.9 lines 1276-1279 + §7.5 lines 1698-1701 explicitly state SO_PEERCRED/host-access is NOT a defense factor. The decomposition spec did not amend those sentences in its master-spec-amendments table. Procedure: for every "factor", "gate", or "defense" claim in a decomposition spec, grep the master spec for the same noun and verify alignment.
- **Master-spec amendment tables miss schema additions for new audit-event shapes.** Decomposition specs that introduce new audit event types (e.g., `crypto.policy_set`) with new fields (e.g., `prev_hash`, `policy_hash`) routinely amend §11.1 phasing (lifecycle scope) and §10 failure modes (some) but forget §4.6 (audit-event-shapes) where the schema actually lives. Always check whether the new event type has a defined wire shape in master spec §4 (or wherever event shapes live), and require a row that adds it. Seen 2026-05-07 Finding 2: `crypto.policy_set` had `prev_hash`, `policy_hash`, server-start ULID, full snapshot — none anchored in master spec §4.6. Two implementers reading sub-epic D would invent different shapes.
- **"Capability" naming collisions with existing host-side authorization types.** When a spec proposes a new player-attribute grant called "capability" (e.g., `crypto.operator`), check if `internal/command/`, `internal/access/`, or other host-side packages already use "Capability" as a structured type. HoloMUSH has `command.Capability{Action, Resource, Scope}` for command pre-flight authorization — distinct from a player-grant string. Force the spec to name the new construct distinctly ("operator-grant", "crypto-operator allow-list", etc.) or call out the collision explicitly.
- **Generated-binding filename pattern claims need verification.** Specs that mention "ConnectRPC bindings generated alongside `*_grpc.pb.go`" usually mis-describe the layout. HoloMUSH actually generates ConnectRPC bindings into `<svc>v1connect/<svc>.connect.go` (a separate sub-package per service), not `<svc>_connect.go` next to `<svc>_grpc.pb.go`. Always run `rg --files <proto-output-dir> | rg connect` before approving a layout claim.
- **"Pinned recompute composition" claims with typed-struct vs bytes asymmetry.** When a spec generalizes a hash-recompute function from `f(*TypedStruct) []byte` to `f([]byte) []byte` to make it primitive-shareable across multiple chains, walk the typed function's body line-by-line for steps that operate on TYPED STATE before marshal (e.g., `if len(canon.PrevHash) == 0 { canon.PrevHash = nil }` — empty-byte vs nil normalization). Those steps have nowhere to live in a `[]byte → []byte` primitive: bytes-already-marshaled cannot distinguish `[]byte{}` from `nil` once JSON has them as `""` vs `null`. The "preserved by construction" claim fails unless the per-chain Canonicalize function absorbs the normalization (which contradicts an "no per-chain divergence in canonicalization" invariant) or callers are required to normalize before invoking pinned (demoting "by construction" to "by caller discipline"). Seen 2026-05-10 in event-payload-crypto-phase5-sub-epic-e R2: pinned `RecomputeSelfHash([]byte)` claimed to subsume D's `ComputePolicyHash(*PolicySetPayload)` whose PrevHash empty-form normalization (chain.go:50-52) is asserted by D's existing test `TestComputePolicyHashNormalizesEmptyPrevHashToNil`.
- **Subject-prefix migration must trigger ABAC-denial-coverage audit.** When a spec moves an audit-event subject family from `audit.*` to `events.<game>.*` (e.g., to land on JetStream's `events.>` SubjectFilter), the master-spec's ABAC denial line for plugin/character subscribers (typically enumerating `audit.*` literally) no longer covers the migrated family. Always grep master spec for `INV-15|deny.*subscribe|audit\.>` and force the §amendments table to either (a) cite a default-deny source that already covers the new prefix or (b) add an explicit ABAC-denial amendment row. Seen 2026-05-10 in sub-epic E: rekey moved to `events.<game>.system.rekey.*`, master INV-15 still enumerates `audit.*`, no §9 amendment row updated INV-15. (D's spec made the same move for `crypto.policy_set` without an ABAC update — likely a latent gap, not a precedent to follow.)
- **Spec-illustrated example registrations must satisfy the spec's own MUST invariants.** When a spec adds new MUST invariants on a registry/struct (e.g., INV-E27 "every Chain MUST populate ScopeFromPayload"), grep every literal example registration in the same spec to verify each MUST field is populated. Plan authors copy the spec's example verbatim and ship non-compliant code; only the meta-test catches it. Seen 2026-05-10 in sub-epic E R2: §3.7's `var RekeyChain = auditchain.Chain{...}` literal omits both `ScopeFromPayload` (INV-E27) and `SelfHashFieldName` (INV-E28's named per-chain responsibility).
- [workflow_run cannot fire on path-skipped upstream](feedback_workflow_run_skipped_upstream.md) — specs that propose a `workflow_run`-triggered aggregator to restore branch-protection coverage for a `paths-ignore`-skipped upstream are broken by construction (no run exists → no event fires). ## Interfaces and boundaries that recur

- **`.jj/repo` dir-vs-file**: in the main checkout (`default` workspace) it is a directory; in any other jj workspace it is a file containing a relative path back to the main repo. Anything that needs to know "am I in the default workspace" or "where is MAIN_REPO" should use this. Reference implementation lives at `Taskfile.yaml:521-530`.
- **SessionStart hook output**: plain stdout is concatenated as additional context. `bd prime` is the canonical example. JSON `hookSpecificOutput.additionalContext` is the alternative but no in-tree hook uses it.
- **`task` cannot mutate caller's shell `pwd`/env**: any spec that wants "after this task, your shell is in directory X" must be a shell function/wrapper, not a task target. Treat as a hard constraint when reviewing automation specs.
