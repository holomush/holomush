# design-reviewer agent memory

This file accumulates HoloMUSH-specific patterns of good and bad designs
discovered during adversarial design review. Entries are added by the agent
itself after completing a review.

Keep under 200 lines. Curate — don't hoard.

## Common spec weaknesses in this codebase

- [oops vs gRPC code conflation](feedback_oops_vs_grpc_code_conflation.md) — HoloMUSH specs often call oops codes "gRPC codes"; flag wording, do not block if pattern matches existing codebase convention
- [grpc FromError message rewrite on wrapped statuses](feedback_grpc_fromerror_message_rewrite.md) — `st.Message()` on a status reached via `errors.As` is the OUTER err.Error(); flag "verbatim" / "wire-equivalent" claims that ignore this
- **"Mirrors existing technique X" claims need verbatim verification.** When a spec says "use the same technique as `Y`", read both sides and diff them. Authors paraphrase or reconstruct from memory and produce structurally different (broken) variants. Seen 2026-04-25 in session-workspace-isolation: spec claimed to mirror `Taskfile.yaml:521-530` gowork's `.jj/repo` resolution, but the provided snippet computed a different path that failed when invoked from a worktree.
- **Shell-snippet error handling in spec docs is rarely verified.** Patterns like `var=$(cmd | tail -n 1) || return` (bash) and `set var (cmd | tail -n 1); or return $status` (fish) do NOT detect failure of `cmd` — the pipeline status is the LAST command's status. Run the snippet before believing it. Seen 2026-04-25.
- **Spec "verifiable by …" criteria with overly-broad jj revsets.** `jj log -r 'all()'` returns full history; for experiment-level verification prefer `mutable()` or scoped revsets. Not blocking but degrades the diagnostic value.
- **Stated rationale for shell idioms can be wrong even when the idiom works.** A spec may correctly include a `cd` or `cd && cmd` block but provide an incorrect *reason* (e.g., "needed because X requires Y" where Y is false). This is documentation rot risk, not a correctness blocker. Verify rationale claims by running the inner command without the `cd` to see if it really fails. Seen 2026-04-25 in session-workspace-isolation r3: spec claimed `cd "$MAIN_REPO"` was needed for `jj git fetch`, but `jj git fetch` works from any workspace because it resolves repo storage via `.jj/repo`.
- [Check existing RPCs before "new RPC" proposals](feedback_check_existing_rpcs.md) — auth/session/check specs often duplicate existing RPCs (e.g., `WhoAmI` vs `WebCheckSession`); always grep proto + handlers
- [Extending an existing RPC: audit callers for contract-shape dependencies](feedback_extending_rpc_check_callers.md) — when a spec extends an RPC's failure-path contract (error → `authenticated=false`), grep all current callers; web client try/catch patterns often depend on the throw
- **"ctx carries an actor at boundary X" claims need a full call-graph trace.** Specs that propose authenticating an actor pulled from `core.ActorFromContext(ctx)` at a specific boundary MUST be checked against EVERY production caller of that boundary, not the obvious one. `core.WithActor` is often called *after* the boundary call returns (for use in a downstream `EmitPluginEvent`), not before. Seen 2026-04-25 in plugin-actor-claim-auth: spec's §3.3.4 issuance flow assumed `Host.DeliverEvent`/`DeliverCommand` ctx carries an actor, but `subscriber.go:104-150` and `dispatcher.go:118-310` both stamp the actor only after the Deliver call. The spec's "verbatim cascade preserved" branch was dead code at every production call site.
- **Migration scope: enumerate ALL in-tree plugins that hit the new gate, not the marquee one.** Specs adding a manifest gate often name one plugin (the binary one with provenance) and miss Lua plugins that emit through a different code path. Always check BOTH `events:` declarations in `plugin.yaml` (subscriber-driven emits) AND command capability `action: emit` (dispatcher-driven emits) across every `plugins/*/plugin.yaml`, then read the corresponding `main.lua` / `main.go` to confirm whether character-driven emits actually happen. Seen 2026-04-25 in plugin-actor-claim-auth: r1 named only `core-scenes` and missed `core-communication`; r2 named both but missed `echo-bot` (Lua plugin subscribed to `events: [say]` that emits a follow-up `say` for non-plugin actors).
- **In-tree manifest filename is `plugin.yaml`, NOT `manifest.yaml`.** The `Manifest` Go type's docstring confirms (`internal/plugin/manifest.go:69, 291`). Specs that say "update `<plugin>/manifest.yaml`" are referring to a path that doesn't exist; the acceptance criterion is unverifiable as written. Always grep `internal/plugin/manifest.go` for the canonical filename before approving any acceptance criterion that names the manifest path.
- **Lint heuristics described in spec acceptance criteria need a verbatim 3-plugin walkthrough.** When a spec proposes "lint check that flags any plugin with property X", apply the proposed predicate to each in-tree plugin yaml in your head and check the result. Seen 2026-04-25 in plugin-actor-claim-auth r3: criterion 8's heuristic checked `action: emit` capabilities OR non-empty `events:`; this caught `core-communication` (capabilities) and `echo-bot` (events:) but silently passed `core-scenes` (top-level `emits: [scene]`, but command capabilities are `action: write` and no `events:`). The spec's listed migration scope had core-scenes in it but the proposed lint check would not enforce its continued correctness. The fix surface is "evaluate the proposed lint heuristic against every plugin in the migration scope" — if any plugin in the scope is not flagged by the heuristic when it should be, the heuristic is wrong, not the scope.
- **Migration-section consistency: every section that lists migrating plugins must list ALL of them.** Specs frequently fix a "missed plugin" finding in the design section but forget the same fix in §migration-path / §release-notes / §risks. Round 2 of plugin-actor-claim-auth surfaced echo-bot as missed; round 3 fixed §3.2, §6, criterion 8 but left §9 step 1 still naming only `core-scenes`. Always cross-grep the spec for `core-scenes` (or whichever was the marquee name) AFTER any migration-scope edit, and ensure each occurrence either references the full list or correctly scopes itself.
- **Lint heuristic enumeration claim ≠ heuristic logic.** When a spec writes "this heuristic catches X via path A and Y via path B and Z via path C", the FAILURE mode is that the conjunction in the heuristic doesn't actually evaluate to true on Z's manifest. Re-evaluate the heuristic predicate term-by-term against EACH plugin in the enumeration's actual on-disk yaml — do not trust the prose. Round 4 of plugin-actor-claim-auth: criterion 8's heuristic required `emits:` non-empty AND `(commands: OR events: non-empty)` AND missing-character-claim; the spec's own enumeration claimed it caught echo-bot "via events + emits in handler" but echo-bot's `plugin.yaml` has no top-level `emits:` field at all (the emits happen at runtime in main.lua), so the conjunction short-circuits to false. The heuristic operates on manifest static state, not runtime Lua handler behavior. When evaluating "does heuristic H catch plugin P", read P's plugin.yaml and check each conjunction term in the actual on-disk manifest — never accept the spec author's enumeration without verifying each entry.
- [Taskfile shell idioms must be verified against mvdan/sh](feedback_taskfile_shell_idioms.md) — go-task's inline `cmd:` runs under `mvdan.cc/sh/v3/interp`, not `/bin/sh`. `exec N>FILE` for N>=3 is rejected by mvdan/sh; fd inheritance into subprocess `task other-task` calls is fds 0/1/2 only (no `ExtraFiles`). Verify empirically (one-Taskfile fixture) before approving any spec with shell tricks in Taskfile cmds. Seen 2026-04-26 in pr-prep-concurrency-safety: spec proposed `exec 9>"$LOCK"; flock -n 9; task pr-prep:run` — empirically fails on first line.
- [go-task `internal: true` semantics](feedback_go_task_internal_semantics.md) — internal tasks are CLI-blocked by go-task itself (any `task <name>` invocation, even from a subshell, exits 202 with "Task is internal") AND hidden from both `--list` and `--list-all`. Layering a shell-level "bypass guard" on top is dead code: go-task's gate fires first. Seen 2026-04-26 in pr-prep-concurrency-safety r2: harness calls `exec task pr-prep:run` from `sh -c` block, but `pr-prep:run` is `internal: true` — lock acquires, info written, body never runs. Plus invariant I-10 asserted "MUST appear in --list-all" which is structurally false for internal tasks.
- [go-task wraps user exit codes by default](feedback_go_task_exit_code_wrapping.md) — go-task collapses any user-cmd non-zero exit to 201 ("Command execution error") unless `--exit-code`/`-x` is passed at every nested invocation layer. Specs that assert literal exit-code passthrough invariants ("harness exits 42 when stub exits 42") will fail. Seen 2026-04-26 in pr-prep-concurrency-safety r3: I-6 propagates_exit_code unsatisfiable as drafted, plus two test-row assertions of "exit 1" that empirically observe exit 201.
- **"Existing field" claims about proto must be verified against the actual `.proto` file, not against comments or adjacent code.** Specs have called proto fields "existing" because a column with the same name exists on a database table or a header with that name exists. They are not the same wire-format surface. Always grep the proto. Seen 2026-04-25 in event-payload-crypto-design: spec marked `codec` as "EXISTING" on the `Event` proto; only the audit table column and NATS header have that name, the proto Event has no codec field.
- **Specs that build on a substrate spec must explicitly state which substrate abstractions they supersede.** When a new design introduces types that overlap or replace abstractions defined and shipped in a prior spec (e.g., `KeyProvider`/`KeySelector` from one spec, replaced by `Provider`/`DEKManager` in the next), the new spec must call out the supersession. Otherwise the implementation has two competing models in the same package. Seen 2026-04-25 in event-payload-crypto-design vs jetstream-event-log-design.
- **"Before X returns" / "synchronously" / "atomically" claims for cross-replica operations need a specified protocol.** NATS core publish, JetStream publish-with-ack, and request-reply have different consistency guarantees. A spec that says "MUST invalidate caches in all replicas before the operation returns" without specifying the ack mechanism is asserting an unprovable invariant.
- **Global negatives ("MUST NOT be written to disk anywhere ever") tagged as "Unit + Lint" are usually too vague to enforce.** Unit tests cannot prove a global negative; lint rules need concrete forbidden-symbol/forbidden-call lists. Force the spec to enumerate the actual checks.
- **Invariants whose AAD or signature includes proto-marshaled bytes must specify the canonicalization rule.** Proto serialization is not deterministic by default; if both encrypt-side and decrypt-side don't use the same canonical encoder a subtle library upgrade silently breaks decryption. Spec must name the canonicalization function.
- **Schema additions referenced in spec body but not in the data-model section.** When a spec mentions new columns on existing tables in flow diagrams or invariants but its data-model section only adds a NEW table, the migration for the additions is undefined. Plan-writers will invent column types. Seen 2026-04-25 in event-payload-crypto-design: `events_audit.dek_ref/dek_version` referenced but never declared in §4.
- **Lifecycle invariants asserted as MUST that the AuthGuard cannot satisfy.** When a spec promises "previous-tenure player retains decryption" but the access decision tree only matches the current binding, the invariant is unprovable. Walk every assertion against the access-control branches before believing it.
- **Fix-text in revisions can introduce NEW factual errors about repo symbols/files.** When a spec author addresses a "list the affected call sites" finding in round N, the resolution paragraph may name additional symbols (e.g., "and also delete `IdentityKeyProvider`") or wrong file locations (e.g., "from `codec.go`" when the symbol is actually in `publisher.go`). Always re-verify the fix-text against `rg` even when the prior round's finding was about the same area. Seen 2026-04-30 in event-payload-crypto-phase2-substrate r2: round 1 asked the author to list `KeyProvider`/`KeySelector`/`KeyLabel` call sites; round 2 added a "Phase 3 scope addition" paragraph that named `IdentityKeyProvider` (does not exist in repo) and located `identityKeySelector` in `codec.go` (it lives in `publisher.go:392`).
- **"Compile-time enforced" claims about proto serialization need to match the actual ruleguard scope.** A spec may claim "bare `proto.Marshal` is compile-time prevented by ruleguard rule X" when X actually only fires for arguments of a specific type unrelated to the proto message in question. Trace the ruleguard predicate to confirm what the rule actually catches. Seen 2026-04-30 in event-payload-crypto-phase2-substrate r2: claim that `dek_no_serialize` ruleguard prevents bare `proto.Marshal(event.Actor)` — but that rule's `Type.Is(...)` filter is on `*dek.Material`/`dek.Material`, not `*Actor`, so it would not fire. The defense is the runtime byte-equality test, not compile-time.
- [golangci-lint module-plugin package name claim](feedback_module_plugin_package_name.md) — module plugins do NOT require `package main`; canonical upstream example uses `package linters`. Verify against `golangci/plugin-module-register/register.go` and `golangci/example-plugin-module-linter`.
- **Lint-rule scope migrations need to account for the OLD rule's de-facto exclusions.** A rule that runs through `gocritic` inherits gocritic's `_test.go` exclusion in `.golangci.yaml`. Migrating that rule to a top-level linter in `linters.enable` LOSES the exclusion unless explicitly re-added. Always grep for callsites in `_test.go` to estimate the regression surface. Seen 2026-05-01 in go-analysis-migration: spec's `ulidmakeforbidden` declared "Allowlist: none (repo-wide)" but the codebase has 1300+ `ulid.Make()` callsites in `_test.go` that the existing rule (via gocritic) silently lets pass.
- [golangci-lint v1 vs v2 exclusion schema](feedback_golangci_v1_vs_v2_schema.md) — `.golangci.yaml` v2 uses `linters.exclusions.rules`, NOT v1's `issues.exclude-rules`. Always check the file's `version:` line before approving any exclusion-block YAML in a spec. Seen 2026-05-01 round 2 of go-analysis-migration: fix-snippet for the `_test.go` ulidmakeforbidden exclusion used v1 syntax against a v2 config, defeating the very fix it was meant to add.

## Interfaces and boundaries that recur

- **`.jj/repo` dir-vs-file**: in the main checkout (`default` workspace) it is a directory; in any other jj workspace it is a file containing a relative path back to the main repo. Anything that needs to know "am I in the default workspace" or "where is MAIN_REPO" should use this. Reference implementation lives at `Taskfile.yaml:521-530`.
- **SessionStart hook output**: plain stdout is concatenated as additional context. `bd prime` is the canonical example. JSON `hookSpecificOutput.additionalContext` is the alternative but no in-tree hook uses it.
- **`task` cannot mutate caller's shell `pwd`/env**: any spec that wants "after this task, your shell is in directory X" must be a shell function/wrapper, not a task target. Treat as a hard constraint when reviewing automation specs.
