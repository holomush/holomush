# Tag-Only Release; GitHub Release Notes as Canonical History

**Date:** 2026-05-24
**Status:** Accepted
**Decision:** holomush-jfb9x
**Deciders:** HoloMUSH Contributors

## Context

release-please maintains an in-repo `CHANGELOG.md`, which requires a commit to
the protected `main` branch on every release. That protected-branch write is the
root cause of a cascade of clunkiness hit during the first real release campaign:

- The `GITHUB_TOKEN`-created release PR does not trigger required CI, so every
  release needed a manual close+reopen.
- The generated `CHANGELOG.md` violated the repo's rumdl rules, requiring
  `CHANGELOG.md` excludes across three Taskfile rumdl flags.

GoReleaser already generates GitHub Release notes today via its `changelog:`
block (with `docs:`/`test:`/`chore:`/merge exclude filters), so the release-notes
surface — the actual must-have — is already covered independent of release-please.
The in-repo `CHANGELOG.md` is the redundant, costly artifact.

## Decision

Cutting a release MUST NOT create a commit on `main`. Version state lives in git
tags only (`cog bump --auto --disable-bump-commit` creates a tag and nothing
else), and **GitHub Release notes are the canonical per-version history**,
generated by GoReleaser. No in-repo `CHANGELOG.md` is maintained.

## Rationale

- The `CHANGELOG.md` commit-to-protected-`main` is the root cause of the
  release-PR mechanism and every downstream CI-trigger failure. Removing the file
  removes the entire class of pain.
- GoReleaser's native notes already exist and already filter `docs:`/`test:`/
  `chore:`. Passing `--release-notes` to feed it an external (e.g. `cog`-rendered)
  changelog would *bypass* those filters, silently bloating notes — so the
  release step MUST NOT pass `--release-notes`.

## Alternatives Considered

**Option A: Keep in-repo `CHANGELOG.md` (release-please model)**

| Aspect     | Assessment                                                                                                                   |
| ---------- | ---------------------------------------------------------------------------------------------------------------------------- |
| Strengths  | Browsable changelog in the source tree; standard OSS convention; works offline / survives leaving GitHub                     |
| Weaknesses | Forces a commit to protected `main` → release-PR that `GITHUB_TOKEN` can't trigger CI for; violates rumdl rules; dual source |

**Option B: Tag-only; GitHub Release notes as canonical history (chosen)**

| Aspect     | Assessment                                                                                                          |
| ---------- | ------------------------------------------------------------------------------------------------------------------- |
| Strengths  | No protected-`main` write; eliminates the release-PR mechanism and rumdl excludes; single notes source (GoReleaser) |
| Weaknesses | Release history is not in the source tree; depends on GitHub as the history host                                    |

## Consequences

**Positive:**

- Eliminates the protected-branch write and the release-PR mechanism entirely.
- Removes the three `CHANGELOG.md` rumdl `--exclude` entries from `Taskfile.yaml`.
- Invariants INV-3 (no protected-main write) and INV-7 (notes preserved) become
  directly testable.

**Negative:**

- Release history is only in GitHub Releases, not the source tree.

**Neutral:**

- GoReleaser's existing `changelog:` config and exclude filters are unchanged.
