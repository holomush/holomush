## Summary

The stated testing discipline (TDD, ACE naming, table-driven tests, Ginkgo/Gomega tiers, the invariant-binding ratchet, quarantine-with-open-issue) is **real in the parts that are mechanically enforced** — lint gates (sloglint, depguard, the 11-analyzer gorules crypto/ULID-misuse plugin), the whole-system plugin census tier, the quarantine bijection meta-test, and the invariant false-green guard are all genuinely wired and, on spot-check, doing real work. A scripted sweep for the "fake-tests-itself" / zero-assertion pattern the project's own plan-review learnings warn about turned up nothing genuine in the higher-risk packages checked — every candidate resolved to a shared assertion helper, a strict mockery mock, or `analysistest`/reflection-based scaffolding.

The one claim that does **not** hold is the headline one: **>80% coverage is not a per-package MUST that CI enforces.** `codecov.yml` only defines project-wide and patch-diff targets (no per-package/component breakdown), and — more importantly — none of the three Codecov status checks are in the branch-protection ruleset's required list at all, so a red Codecov result never blocks a merge. This is not theoretical: the PR immediately preceding this review's baseline commit merged to `main` with Codecov reporting 54.6% patch coverage (❌, its own target is 80%), and `cmd/holomush`'s live package coverage measured this session is 53.4%, both well under the "MUST maintain >80%... Per-package" language in `CLAUDE.md`/`testing.md`.

**Counts:** High 1, Medium 1, Low 1, Strengths 9.

## Findings

### HIGH-1 The ">80% per-package coverage" MUST is not enforced by CI, and main currently violates it

- **Severity:** High
- **Claim:** `CLAUDE.md:187` and `.claude/rules/testing.md:25` both state "MUST maintain >80% coverage | Per-package coverage must exceed 80%", but (a) no per-package coverage check exists anywhere in the repo, (b) the only coverage gate that exists (Codecov's project/patch targets) is not a required branch-protection check, and (c) the current `main` branch demonstrably fails both the per-package and the patch bar.
- **Evidence:**
  - `codecov.yml:12-19` defines only `coverage.status.project.default` (target 80%, threshold 2%, i.e. repo-wide aggregate) and `coverage.status.patch.default` (target 80%, threshold 5%, i.e. only newly-changed lines in a diff) — there is no `flags:`/`component_management:` section, so no package or directory is broken out individually. Confirmed via `rg -n "flags:|component_management" codecov.yml` → zero matches.
  - `gh api repos/holomush/holomush/rulesets/11923801` (ruleset `protect-main`, active) lists `required_status_checks.required_status_checks` as exactly: `Build`, `Lint`, `Test`, `CodeRabbit`, `Integration Test`, `E2E Test`. No `codecov/project` or `codecov/patch` context appears. `gh api repos/holomush/holomush/branches/main/protection` independently confirms classic branch protection is unused (404 "Branch not protected") — the ruleset above is the sole enforcement surface.
  - `.github/workflows/ci.yaml:140-144,182-187,249-254` uploads coverage via `codecov/codecov-action` with `fail_ci_if_error: false` on all three jobs (unit/integration/e2e) — even the upload step itself is best-effort, and its result feeds a check that isn't required regardless.
  - Live proof this isn't hypothetical: `gh pr view 4782 -R holomush/holomush --json comments` (PR #4782, "feat(eventbus): external/clustered NATS deployment... (Phase 3)", which is the commit at `HEAD` of the reviewed baseline `30d55a162`) shows Codecov's own comment: "❌ Patch coverage is `54.64231%` with `298 lines` in your changes missing coverage", with `cmd/holomush/cmd_audit.go` at 23.58% patch coverage (159 missing lines) — none of `cmd_audit.go`, `internal/eventbus/audit/replay.go`, or `internal/eventbus/scopecheck.go` (the three lowest-covered files listed) are in `codecov.yml`'s `ignore:` exemption list. The PR merged anyway.
  - Independently measured this session (`go test -cover ./cmd/holomush/...`, run at repo root, `chore/arch-review` branch = `origin/main`): `ok github.com/holomush/holomush/cmd/holomush 7.042s coverage: 53.4% of statements` — the whole `cmd/holomush` package sits at 53.4%, not 80%, on `main` today. (`internal/eventbus` alone, same method, measures 78.1% — closer, but still under the stated bar, and that number excludes the `-coverpkg=./...` cross-package attribution `task test:cover` uses, so it is a conservative lower bound, not the final word.)
- **Impact:** Contributors and reviewers who read `CLAUDE.md`'s testing table take away a false guarantee: "coverage can't drop below 80% per package because CI would fail." In practice a PR can land with a brand-new ~700-line file (`cmd_audit.go`) at 23.58% coverage and nothing red-flags it at the merge gate — the Codecov PR comment is purely advisory. This erodes the specific thing this review dimension is asked to check (coverage *honesty*): the number in the docs and the number CI actually gates are different metrics (per-package vs. project-wide-aggregate-with-2%-slack, and neither is required), and the gap is currently being used.
- **Recommendation:** Either (a) add `codecov/project` and `codecov/patch` (or their per-check names) to the `protect-main` ruleset's required status checks so the existing 80% targets actually gate merges, or (b) if per-package granularity is the real intent, add `component_management`/`flags` to `codecov.yml` scoped to the packages that matter most (`internal/access`, `internal/eventbus/crypto`, `internal/plugin`) and make those required — plus a cheap `go tool cover -func=coverage.out | awk` per-package floor script wired into `task lint` or `task pr-prep` for immediate local feedback, mirroring how `lint:test-helpers` already does a grep-based structural check. Whichever path, update `CLAUDE.md`/`testing.md` to describe the metric CI actually gates today, so the documented bar and the enforced bar agree.
- **Dedup:** none (checked open-issues.json for `codecov`/`per-package`/`coverage gate`/`required check`/`branch protection` — no match; #4631 "codecov debt" is about backfilling specific Phase-5 tests, not the enforcement-mechanism gap itself).

### MEDIUM-1 `quarantine:audit` (INV-3, "flag rows whose GitHub issue is closed") is never invoked automatically

- **Severity:** Medium
- **Claim:** `scripts/quarantine-audit.sh`, wired as `task quarantine:audit` (`Taskfile.yaml:251-254`), is the only mechanism that detects a quarantine row whose tracking issue has since closed (i.e., a test that should have been un-quarantined). It is not called by `task lint`, `task pr-prep`, `task pr-prep:full`, any `deps:`/`cmds: [- task: ...]` reference in `Taskfile.yaml`, or any `.github/workflows/*` job.
- **Evidence:** `rg -n "quarantine" Taskfile.yaml` returns only the task's own definition (lines 251-254) — no other task references it. `rg -n "quarantine" .github/workflows/*.yaml .github/workflows/*.yml` shows quarantine markers used only as a Playwright `--grep-invert @quarantine` filter (`ci.yaml:246`) and as a `continue-on-error: true`, non-gating health probe in `nightly-soak.yml:83-116` (which runs the quarantined specs themselves, not the audit script). Today's registry is small and clean (4 entries, all cross-checked live against `gh issue view` state = OPEN as of this session — `test/quarantine.yaml` issues 4613/4598/4698/4605), so there is no current live instance of a stale row; the gap is structural, not an active incident.
- **Impact:** A quarantined test whose underlying bug gets fixed (issue closed) but whose `Skip`/`quarantinetest.Enabled()` marker is never removed continues silently excluded from the gating suite indefinitely — nobody is prompted to notice unless they manually run `task quarantine:audit` (which requires local `gh` auth). Low current blast radius given the registry's small, healthy state, but the enforcement described by the script's own docstring ("Fail if any quarantined spec's tracking GitHub issue is closed... INV-3") doesn't fire anywhere automatically, so INV-3 is asserted in prose but not actually gated.
- **Recommendation:** Add `task quarantine:audit` as a step in `nightly-soak.yml`'s existing `quarantine` job (which already checks out the repo and has `gh`-capable context via `GITHUB_TOKEN`), or fold it into `task lint` behind a `gh`-available guard (the script already no-ops gracefully — `scripts/quarantine-audit.sh:14-17` — when `gh` isn't on PATH, so adding it to `task lint` would be safe in environments without `gh`).
- **Dedup:** none

- **Recommendation:** No action needed now; if it's ever worth confirming the mechanism works before it's needed for real, a one-off manual `workflow_dispatch` run once a test *is* quarantined would validate the wiring. Not worth a standing fix.

## Strengths

- **The whole-system plugin census tier is real, not a stub.** `test/integration/wholesystem/census_test.go:31-88` drives `integrationtest.Start(suiteT, integrationtest.WithInTreePlugins())` — the actual production `Manager.LoadAll` path — and asserts all 9 registrable in-tree plugins load, commands register, and the late-bound command querier enumerates real commands. The bijection meta-test (`test/meta/quarantine_registry_test.go:35-47`) walks every `_test.go`/`.spec.ts` for marker-shaped lines and asserts the marker set exactly equals the registry set — a genuine structural guard, not decorative (confirmed the F-E12 case at `cmd/holomush/admin_read_stream_e2e_test.go:831-835` gates via a real `if quarantinetest.Enabled()` conditional, not just a comment).
- **The invariant false-green guard is real and has already caught the exact failure mode it's designed for.** `TestBoundInvariantsAreGenuinelyAsserted` (`test/meta/invariant_registry_test.go:1128-1203`) walks every `binding: bound` entry's `// Verifies:` sites and fails if every site is a Skip-only placeholder. Spot-checking the historical example this file's own learnings doc cites (INV-PRIVACY-6) confirms it: the registry entry (`docs/architecture/invariants.yaml`, id `INV-PRIVACY-6`) is correctly `binding: pending` today with an inline comment explaining exactly which clause is untested — the partial-binding bug was found and fixed, not still latent.
- **Zero-assertion / tautological-test sweep came back clean in the packages checked.** A scripted scan for `func TestX(t *testing.T)` bodies with no assertion-shaped call flagged ~130 candidates; after broadening the assertion pattern to account for the project's own helper idioms and manually reading the remainder (~40, concentrated in `internal/plugin/hostcap`, `internal/eventbus`, `internal/access/policy/attribute`, `gorules/analyzers`, `plugins/core-communication`), every one resolved to a genuine (if indirect) assertion: shared helpers like `requireOpaqueInternal`/`requireWorldNotFound` (`internal/plugin/hostcap/{session,world}_test.go`) that assert the exact opaque-error contract from `.claude/rules/grpc-errors.md`; `requireOopsCode` (`internal/eventbus/scopecheck_test.go:18-25`) asserting top-level oops codes per the same rule's documented pattern; strict-mockery-mock "the assertion is that nothing unexpected was called" tests (documented inline, e.g. `internal/auth/auth_service_test.go:303`); `analysistest.Run` fixture-based assertions in every `gorules/analyzers/*_test.go`; and a reflection-based field-completeness parity guard (`internal/plugin/emit_intent_parity_test.go:19-65`) that fails the build the moment a new `EmitIntent` field is added without wiring both the Lua and binary construction sites.

- **The production/test-boundary depguard rule matches its documentation exactly.** `.golangci.yaml:136-155` denies `internal/eventbus/eventbustest`, `internal/core/coretest`, `internal/testsupport/quarantinetest`, and `internal/testsupport/natstest` imports from non-test, non-testsupport files — precisely the four packages `.claude/rules/testing.md` documents as production-import-forbidden, with matching holomush-issue citations in the `desc:` fields.
